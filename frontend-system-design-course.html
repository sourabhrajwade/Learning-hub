Module 1 · Foundations Beginner

1.1 What is Frontend System Design?

Frontend system design is the discipline of architecting client-facing applications so they're fast, scalable, maintainable, secure, and accessible. Unlike backend system design (which focuses on services, databases, and throughput), frontend design centers on the user's device: rendering, network constraints, state, UX latency, and team scaling.

A frontend system designer answers questions like:

  • How does this UI render under 3G on a low-end Android?
  • How do we handle 100k items in a list without freezing the browser?
  • How do 50 engineers ship to one app without breaking each other?
  • How do we keep secrets safe when all our code ships to the user?
Mental model: Treat the browser as a constrained, hostile, single-threaded device on a flaky network owned by an untrusted user. Design accordingly.

Q: How do you scope a 45-min frontend system design question like "Design Instagram"?
Spend ~5 min on requirements (feed, stories, upload? — pick 1-2 core flows), ~5 on high-level architecture, ~10 on data model & API, ~15 on a chosen deep-dive (e.g., infinite scroll + media handling), and ~5-10 on optimizations and trade-offs. Always state assumptions out loud.

1.3 Browser, HTTP, and Networking Essentials

Every FE designer must understand the request lifecycle:

  1. DNS lookup → resolve domain to IP (cached at OS/browser).
  2. TCP + TLS handshake → ~1-3 RTTs (HTTP/2) or ~1 RTT (HTTP/3 / QUIC with 0-RTT resumption).
  3. HTTP request → server returns HTML.
  4. Critical resource discovery → browser parses HTML and requests CSS, JS, images.
  5. Render → DOM + CSSOM → Render Tree → Layout → Paint → Composite.

1.4 The Browser Rendering Pipeline

HTML ──► DOM ─┐
                 ├─► Render Tree ─► Layout ─► Paint ─► Composite ─► Pixels
CSS  ──► CSSOM ─┘                       ▲           ▲
                                         │           └─ GPU layers
JS  (can mutate DOM/CSSOM, blocks parser)
  • Layout (reflow): expensive; triggered by geometry changes (width, top, font-size).
  • Paint: filling pixels; triggered by color, background, visibility.
  • Composite: cheap; transform and opacity only — animate these for 60fps.
Example: Animating left: 0 → 100px triggers layout on every frame. Use transform: translateX(100px) instead — composite-only, GPU-accelerated.
Q: Why does adding will-change: transform sometimes hurt performance?
It promotes the element to its own GPU layer, consuming memory. Overuse causes layer explosion and can increase memory pressure and paint times. Only apply it just before an animation, then remove it.

2.4 Data Fetching & Caching

Patterns

  • Fetch-on-render: simple, but waterfalls. Component renders → effect fires → fetches.
  • Render-as-you-fetch: kick off requests during route transition (React Router loaders, Next.js).
  • Stale-while-revalidate: show cached data instantly, refetch in background.
  • Optimistic updates: mutate UI before server confirms; roll back on error.
// TanStack Query: SWR + optimistic update
const mutation = useMutation({
  mutationFn: likePost,
  onMutate: async (postId) => {
    await queryClient.cancelQueries(['post', postId]);
    const prev = queryClient.getQueryData(['post', postId]);
    queryClient.setQueryData(['post', postId], (p) => ({...p, liked: true, likes: p.likes+1}));
    return { prev };
  },
  onError: (_err, postId, ctx) => queryClient.setQueryData(['post', postId], ctx.prev),
  onSettled: (_d, _e, postId) => queryClient.invalidateQueries(['post', postId]),
});

2.5 Component Design Principles

  • Composition over configuration — prefer <Card><Card.Header/></Card> over <Card hasHeader headerText="..."/>.
  • Props as a contract — narrow, discriminated unions over boolean soup.
  • Headless components (Radix, React Aria) — separate behavior from style.
  • Controlled vs uncontrolled — controlled when you need to enforce invariants.
Q: Design a reusable <Modal> component. What are the key concerns?
Focus trapping, restore focus on close, ESC handling, click-outside, scroll lock, portal-based rendering, ARIA role="dialog" + aria-modal="true", controlled/uncontrolled API, animation hooks, stacking multiple modals, SSR safety. Use a headless lib (Radix/Aria) unless you have a strong reason not to.
Q: When would you use Context vs Redux vs Zustand?
Context: low-frequency, tree-scoped values (theme, locale, current user). Re-renders all consumers on change — bad for hot state. Redux: large apps with strict patterns, time-travel debugging, middleware ecosystem. Zustand: minimal boilerplate, selector-based subscriptions, great default for medium apps.

3.2 Bundling & Code Splitting

  • Route-based splitting: each route is a chunk (React.lazy + Suspense, or framework router).
  • Component-level: heavy components (charts, editors) loaded on demand.
  • Vendor splitting: separate react-dom, etc., for long-term caching.
  • Tree-shaking: ESM + sideEffects:false; avoid barrel files that import everything.
  • Module federation: share runtime deps across micro-frontends.
// Route-level split
const Settings = React.lazy(() => import('./Settings'));
<Suspense fallback={<Skeleton/>}><Settings/></Suspense>
Watch out: A single import like import _ from 'lodash' pulls in ~70 KB. Use import debounce from 'lodash/debounce' or lodash-es.

3.3 Images, Fonts, and Assets

Images

  • Modern formats: AVIF > WebP > JPEG; serve via <picture> with fallbacks.
  • Responsive: srcset + sizes to ship right resolution per device.
  • loading="lazy" below-the-fold; fetchpriority="high" for LCP image.
  • Reserve space with width/height to prevent CLS.

Fonts

  • Self-host (avoid third-party DNS hops); font-display: swap; preload the critical face.
  • Subset to needed glyph ranges (latin, latin-ext).
  • Use variable fonts to ship 1 file for many weights.

3.4 Runtime Optimization

  • Memoization: React.memo, useMemo, useCallback — only when measured. Default React renders are cheap.
  • Concurrent features: useTransition to mark non-urgent updates; useDeferredValue for stale-but-responsive UI.
  • Web Workers: offload parsing, search, image processing (Comlink simplifies the messaging).
  • Debounce/throttle: input handlers, scroll, resize.
  • Avoid layout thrashing: batch reads then writes; don't read offsetHeight in a write loop.

3.5 Virtualization & Long Lists

Rendering 10,000 DOM nodes kills the main thread. Virtualization renders only what's visible plus a small overscan buffer.

// react-window
<FixedSizeList height={600} itemCount={10000} itemSize={48} width="100%">
  {({ index, style }) => <Row style={style} item={items[index]}/>}
</FixedSizeList>

Use case: Slack message list, Gmail inbox, Excel-like grids. For variable heights use VariableSizeList or @tanstack/react-virtual.

Q: Your dashboard is janky on scroll. How do you debug?
Open DevTools Performance tab, record a scroll. Look for long tasks (>50ms). Identify if it's scripting (JS), rendering (layout/paint), or compositing. Common fixes: virtualize the list, move expensive work off the main thread, replace layout-triggering animations with transform/opacity, reduce React re-renders via memoization or selectors.
Q: How do you reduce a 2 MB JS bundle?
(1) Audit with source-map-explorer or webpack-bundle-analyzer. (2) Code-split by route. (3) Replace heavy deps (moment → date-fns/Temporal, lodash → tiny utils). (4) Lazy-load below-the-fold features. (5) Modern targets only — drop legacy transpilation for evergreen browsers. (6) Compression (Brotli) at the CDN.

Module 4 · CDNs (Deep Dive) Advanced

4.1 What & Why

A Content Delivery Network is a globally distributed network of edge servers (PoPs — Points of Presence) that cache and serve content close to users. Goals: reduce latency, offload origin, improve availability, and provide a security layer (DDoS, WAF, bot management).

How it works

  1. User makes request; DNS (often Anycast) routes to the nearest PoP.
  2. Edge checks its cache. Hit → served in ~10-30ms. Miss → fetches from regional shield → origin.
  3. Response is cached based on rules (TTL, cache key, headers).

Major players: Cloudflare, Fastly, Akamai, AWS CloudFront, Google Cloud CDN, Vercel/Netlify (built on Cloudflare/AWS).

4.3 Invalidation Strategies

Cache invalidation is famously hard. Three primary approaches:

  1. TTL-based: Set short TTL; staleness bounded by time. Simple, but adds origin load.
  2. Purge: API call (PURGE /path or by surrogate key) invalidates the URL globally. Fastly purges in ~150ms.
  3. Surrogate keys (cache tags): tag each response with keys like product:42; purge by key when product 42 changes — more granular than URL.

Stale-while-revalidate (SWR)

Cache-Control: max-age=10, stale-while-revalidate=60

For 10s, response is fresh. From 10s-70s, edge serves stale instantly while async-fetching fresh. After 70s, blocking fetch. Great for recommendation widgets, "trending" lists, etc.

4.4 Edge Computing

Modern CDNs run code at the edge: Cloudflare Workers (V8 isolates), Fastly Compute@Edge (WASM), Vercel Edge Functions. Use cases:

  • A/B testing & feature flags at the edge — no client round-trip to a flag service.
  • Personalization while keeping HTML cacheable (HTML rewriting / ESI).
  • Geo routing & redirects — send EU users to /eu bucket.
  • Edge SSR / streaming — render closer to the user.
  • Auth/token verification before reaching origin.
  • Image transformation (Cloudflare Images, Imgix-style on the fly).
Q: A user reports they see a stale price on the product page after a price update. Diagnose.
Possible layers caching: browser disk cache, service worker, CDN edge, regional shield, Next.js ISR cache, origin response cache. Check (a) the URL's response headers (cf-cache-status, x-cache), (b) whether your purge included the surrogate key/path, (c) edge KV/data cache TTL, (d) browser cache age. Fix: add a versioned URL or use surrogate-key purge on price update.
Q: How do you cache personalized HTML?
Don't cache the personalized parts. Options: (1) cache shell HTML, hydrate personalized regions client-side; (2) Edge Side Includes (ESI) — cacheable shell with dynamic fragments composed at the edge; (3) HTML rewriting at the edge using Cloudflare's HTMLRewriter; (4) split into two requests — cached static + fast personalized JSON. (5) Use Vary sparingly — wide vary keys (e.g., on user-id) destroy hit rates.
Q: Why is Anycast DNS used by CDNs?
Anycast advertises the same IP from many PoPs; BGP routes the user to the topologically nearest one. Result: low-latency DNS + automatic failover when a PoP goes down, without DNS changes.

Module 5 · Security (Deep Dive) Advanced

5.1 Cross-Site Scripting (XSS)

Attacker injects script that runs in the victim's origin, accessing cookies, DOM, and APIs.

TypeVector
StoredPersisted in DB (e.g., comment) and served to other users.
ReflectedEchoed from URL/query into the page.
DOM-basedClient-side JS writes untrusted input into innerHTML, eval, etc.

Defenses

  • Contextual output encoding — frameworks (React, Vue, Svelte) escape by default. Avoid dangerouslySetInnerHTML and v-html.
  • Sanitize HTML from rich-text editors with DOMPurify.
  • Content Security Policy (see 5.5).
  • Trusted Types (Chrome) — make string-to-DOM sinks fail by default.
  • HttpOnly cookies — JS can't read them, mitigating session theft.
// React: safe by default
<div>{userInput}</div>             // ✅ escaped
<div dangerouslySetInnerHTML={{ __html: userInput }}/>  // ❌ XSS risk
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userInput) }}/> // ✅

5.2 Cross-Site Request Forgery (CSRF)

An attacker site causes the victim's authenticated browser to perform a state-changing request to your site (e.g., transfer money). Cookies are auto-attached on cross-origin requests unless protected.

Defenses

  • SameSite=Lax (default in modern browsers) — blocks cross-site cookies on most cross-origin requests; Strict for sensitive flows.
  • CSRF tokens — synchronizer token pattern; double-submit cookie pattern.
  • Use custom headers (e.g., X-Requested-With) — browsers preflight, so they can't be forged from a plain HTML form.
  • Avoid GET for state changes.

5.3 Same-Origin Policy & CORS

Origin = scheme + host + port. SOP blocks JS from one origin reading responses from another. CORS is a controlled relaxation: the server opts in via headers.

// Server response for cross-origin allowed
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Never use Access-Control-Allow-Origin: * together with Allow-Credentials: true — browsers reject it, and even if they didn't, it would be a major leak.

5.4 Authentication: Cookies vs JWT in localStorage

HttpOnly Cookie (session)JWT in localStorage
XSSResilient (JS can't read)Vulnerable (any XSS = full token theft)

Recommendation: Use HttpOnly; Secure; SameSite=Lax cookies for session auth on first-party apps. Use short-lived access tokens + refresh in HttpOnly cookie. Avoid storing tokens in localStorage for sensitive apps.

5.5 CSP, SRI, Security Headers

Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-rAnd0m';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://cdn.example.com;
  connect-src 'self' https://api.example.com;
  frame-ancestors 'none';
  upgrade-insecure-requests;
  • CSP: whitelist of allowed sources; nonce/hash for inline scripts.
  • SRI (Subresource Integrity): <script src="..." integrity="sha384-..."> — refuses tampered third-party scripts.
  • HSTS: Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
  • X-Frame-Options / frame-ancestors — clickjacking defense.
  • Referrer-Policy, Permissions-Policy — data minimization.

5.6 Supply Chain

Your node_modules is your runtime. A malicious dependency can exfiltrate everything users type.

  • Lockfiles + reproducible builds (npm ci).
  • Audit (npm audit, Snyk, Dependabot, Socket.dev).
  • SRI for third-party CDN scripts.
  • Minimize transitive deps; prefer std lib / small libs.
  • Run a CSP report-only mode in production to catch unexpected origins.
Q: Your blog allows user comments. Walk through the XSS risk and your defenses.
Risk: stored XSS via comment HTML. Defense layers: (1) Server stores raw input — render with framework escaping (React) so HTML is treated as text. (2) If allowing rich text, sanitize with DOMPurify on render and ideally on save. (3) CSP with strict script-src and nonces. (4) HttpOnly cookies for sessions so token theft is harder. (5) Trusted Types where supported. (6) Periodic security review of any dangerouslySetInnerHTML.
Q: Why is postMessage dangerous?
It crosses origins. If you don't validate event.origin you accept commands from any site that can iframe or open you. Always check origin and treat the message as untrusted input — schema validate, never eval.

Module 6 · Scalability for Large Apps Advanced

6.1 Monorepos

One repo, many packages. Tools: Nx, Turborepo, pnpm workspaces, Bazel, Rush.

  • Shared tooling, atomic cross-package changes, single-version policy.
  • Affected-only builds (Nx affected, Turbo task graph) = fast CI.
  • Remote caching across the team.
  • Trade-off: tighter coupling, complex CI, larger repo. Worth it past ~3-5 collaborating apps.

6.2 Design Systems

  • Tokens (color, spacing, typography) generated to CSS vars / native platforms.
  • Primitives (Button, Input, Modal) — accessible, themable.
  • Patterns (Form, Table, EmptyState) — composed primitives.
  • Docs with Storybook, automated visual regression tests (Chromatic, Percy).
  • Versioned releases; deprecation policy; migration codemods.

6.3 Micro-Frontends

Independent teams ship independent frontends, composed at runtime or build time. Useful when team count > ~5 and release cadences differ.

Composition strategies

  • Build-time: npm packages composed in a shell. Simplest; coupled releases.
  • Server-side composition: shell stitches HTML fragments (Tailor, Podium, Edge Includes).
  • Run-time / Module Federation (Webpack 5): host loads remotes at runtime, sharing React/etc.
  • Iframes: strong isolation; bad for UX (focus, perf, deep links).
  • Web Components: framework-agnostic boundary.
Costs: dependency duplication, version skew, design inconsistency, perf regressions, cross-MFE communication complexity. Don't adopt for small teams.

6.4 Internationalization & Accessibility

i18n

  • ICU MessageFormat for plurals/genders; tools: react-intl, i18next, FormatJS.
  • RTL support: logical properties (margin-inline-start), dir="rtl".
  • Locale-aware sorting, dates (Intl.DateTimeFormat), numbers, currency.
  • Lazy-load locale bundles by route.

a11y

  • Semantic HTML first; ARIA only when needed.
  • Keyboard support: tab order, focus management, skip links.
  • Color contrast (WCAG AA: 4.5:1 text), prefers-reduced-motion.
  • Automated tests: axe-core, Lighthouse; manual: screen reader passes.

6.5 Observability & Releases

  • RUM (Real User Monitoring): web-vitals lib → analytics endpoint.
  • Error tracking: Sentry, Bugsnag — source maps for stack traces.
  • Feature flags: LaunchDarkly, Statsig; gradual rollouts, kill switches.
  • Releases: canary deploys, automatic rollback on regressed CWV/error rate.
  • Synthetic monitoring: Lighthouse CI, Playwright on schedule.
Q: 5 teams want to ship to one app. Do you go micro-frontend?
Not automatically. First try: monorepo + clear module ownership + design system + trunk-based development with feature flags. MFEs only if teams need independent deploys, different tech stacks, or have hard organizational boundaries (e.g., acquisitions). The cost of MFE is real: duplicate deps, runtime contracts, perf overhead.

Module 7 · Real-World Case Studies Advanced

7.1 Twitter / News Feed

Requirements: Infinite-scroll timeline, real-time updates, like/retweet, media, < 1s perceived load.

Key decisions

  • Pagination: cursor-based (?since_id=...&max_id=...). Cursors are stable under inserts; offset isn't.
  • Virtualization for long feeds + dynamic heights (TanStack Virtual).
  • Optimistic updates for like/retweet.
  • WebSocket / SSE for "X new tweets" pill — don't auto-prepend; UX problem.
  • Image strategy: low-quality placeholder (LQIP) → progressive enhancement; fetchpriority="high" for above-the-fold.
  • Cache normalization: store tweets by ID once; lists store IDs only — single update propagates everywhere.

Data model (client cache)

state = {
  tweets: { [id]: TweetEntity },
  feeds: {
    home:    { ids: [...], cursor: '...', loading: false },
    profile_42: { ids: [...], cursor: '...' }
  }
}

7.2 Chat (WhatsApp / Slack)

  • Transport: WebSocket primary, fallback to long-polling. Heartbeats for liveness.
  • Offline-first: IndexedDB store; outbox queue for unsent messages with exponential retry.
  • Ordering & dedup: client-generated IDs (UUID/ULID) replaced by server IDs on ack — keys stay stable.
  • Read receipts & typing: debounced events; coalesce server-side.
  • Reconnection: on resume, send last-known seq number; server replays missed.
  • Scroll UX: virtualization, "scroll to bottom" only when user is near bottom; stickiness.
  • E2E encryption (WhatsApp): keys never leave client; web client mirrors mobile via QR pairing.

7.3 Google Docs (Real-time Collaboration)

Core problem: two users edit simultaneously — how do edits converge?

  • OT (Operational Transform): server transforms ops against concurrent ops; used by Google Docs.
  • CRDTs (Yjs, Automerge): operations are commutative; works peer-to-peer; popular in newer tools.
  • Presence: awareness protocol with cursors and selections.
  • Offline editing: CRDTs shine — merge on reconnect.
  • Persistence: snapshot + op log; periodic compaction.
  • Conflict-free formatting: formatting as ranges with stable identifiers.

7.4 Netflix (Video Streaming UI)

  • Adaptive Bitrate Streaming (HLS / DASH): MPD/manifest lists segments at multiple bitrates; client picks based on bandwidth + buffer.
  • Pre-fetching: next episode metadata + thumbnails; key art preloaded for hovered tile.
  • Skeleton + low-bitrate poster while metadata loads.
  • Personalization at edge — cacheable shell + per-user rows JSON.
  • Memory management: long sessions; lazy unmount off-screen rows; revoke blob: URLs.
  • DRM: EME/Widevine/FairPlay/PlayReady — license server + content keys.
  • Subtitles: WebVTT, accessibility considerations.

7.5 Autocomplete / Typeahead

The classic frontend system design question.

  • Debounce input (~150-300ms) so we don't fire per keystroke.
  • Cancel in-flight requests on new keystroke (AbortController).
  • Race conditions: drop responses for stale queries — track latest query id.
  • Caching: per-prefix LRU; "abc" results contain "abcd" candidates often.
  • Keyboard nav: ↑/↓/Enter/Esc; ARIA role="combobox", aria-activedescendant.
  • Trie/server side when offline-first or low latency required.
const ctrl = useRef<AbortController>();
async function search(q: string) {
  ctrl.current?.abort();
  ctrl.current = new AbortController();
  const data = await fetch(`/search?q=${encodeURIComponent(q)}`,
    { signal: ctrl.current.signal }).then(r => r.json());
  setResults(data);
}

  • Jump into code without architecture.
  • Drop buzzwords without depth ("we'll use Redux"… why?).
  • Ignore the interviewer's hints.
  • Over-engineer: micro-frontends for a TODO app.
  • Forget mobile, slow networks, and edge cases.

Step 1 · Clarify

Vertical scroll, one full-screen video per "page". Auto-play current, mute by default until interaction. Like, comment, share. Mobile + desktop web.

Step 2 · Global delivery (CDN-heavy)

Video is the largest asset; ByteDance operates a massive video CDN (Volcano Engine + commercial CDNs). Anycast routes user to nearest PoP holding HLS/DASH segments. Manifests cached <60s (rotation + popularity sensitivity); segments cached effectively forever (immutable, content-hashed names).

Step 3 · Prefetch pipeline

Critical for sub-100ms scroll-to-play: while video N plays, prefetch (a) next 2 video manifests, (b) first 1-2 segments of next video, (c) poster image of next 3. Use fetchpriority, <link rel=preload> for the next item. Multiple parallel HTTP/3 streams to the CDN.

Step 4 · Player & intersection

One pool of 3 reusable <video> elements (current, prev, next) — creating new <video> per scroll is slow and leaks decoders. IntersectionObserver fires play on entry, pause + reset to 0 on exit. Mute by default; unmute on first user gesture.

Step 5 · Bandwidth adaptation

HLS ABR. On slow connections, start at lowest bitrate; ramp up. Visible video gets all bandwidth; prefetches lower priority. Mobile users on cellular: respect "data saver" toggle.

Step 6 · Interactions

Double-tap to like (animated heart at tap point). Comment drawer slides up — video continues at edge of viewport. Share generates short URL via API. Optimistic like/follow.

Step 7 · Telemetry & ranking

Watch-time signals (play, complete-pct, skip-fast) sent batched. Critical for the recommendation ML loop. Sample carefully — not every micro-event.

Step 8 · Memory bounds

Long sessions = memory growth. Revoke blob: URLs, drop decoded video buffers off-screen, cap LRU at ~5 videos. Watch the heap profile for leaks (decoders are notorious).

Most sections static HTML (no JS). Interactive sections (cart, search) hydrate independently — partial hydration, like Astro Islands. Reduces JS shipped per page dramatically.

Edge function reads experiment cookie, picks variant section ordering, rewrites HTML inline. Variants edge-cached separately. Merchants can A/B test layouts without backend changes.

Per-shop locales; locale bundles served from CDN. Currency switching: cart fragment re-fetched in new currency; product prices updated in place (HTML body unchanged).
