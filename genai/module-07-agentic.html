Module 7 · Agentic AI

Agents, tools, planning loops, and orchestration.

3 Agentic AI

Listed as "a plus" — likely a differentiator. Be ready to talk about agents vs workflows, tool use, and failure modes.

Q1 What is "Agentic AI" and how is it different from a typical LLM workflow?

A workflow is a fixed graph: deterministic steps with LLM calls at known points. An agent decides its own next step — it has tools, a goal, and a loop where it observes, plans, and acts until it stops. Agents trade predictability for flexibility. Use agents when the path can't be enumerated up front; use workflows when it can. Most production systems are workflows with small agentic sub-steps.

Q2 What are the failure modes of agents, and how do you mitigate them?

Looping (calling the same tool forever) — cap iterations and detect cycles. Drift (forgetting the goal) — re-inject the objective, summarize history. Tool misuse — strict schemas, validate args, dry-run destructive actions. Cost runaway — token + tool-call budgets per session. Prompt injection via tool output — treat tool results as untrusted, never let them rewrite the system prompt. Silent failure — log every step, replay traces in a debugger like LangSmith.

Q3 How would you design an agent for a "Client Pathway" use case — say, onboarding a new customer?

I'd not start with an agent. I'd map the pathway as a workflow first: KYC check, doc collection, eligibility, account creation, welcome comms. Then I'd identify the steps that genuinely need flexibility — e.g., handling unstructured client questions or document discrepancies — and put a small, scoped agent there with a narrow toolset (lookup, draft response, escalate). Hard guardrails: agents can read freely but write only via approved actions with human-in-the-loop on high-risk ones.
