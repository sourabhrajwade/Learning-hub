5e LangChain v1 Agents & LangGraph Workflows

Two patterns you must be able to write live: an agent using LangChain v1's create_agent(), and a graph workflow using LangGraph (no agent — just deterministic nodes and edges). Know the difference and when to choose each.

Study Notes — Agent vs Workflow

Agent (LangChain create_agent)Workflow (LangGraph StateGraph)
Control flowLLM decides next step (tool-call loop)You define the graph; deterministic edges
When to useOpen-ended tasks; path can't be enumeratedKnown steps; you want predictability
Cost / latencyVariable, often higher (multiple LLM calls)Predictable; fewer LLM calls
DebuggabilityHarder — model decisions hiddenEasy — every transition is in code
Production fitUse sparingly, with guardrails & HITLDefault for most enterprise GenAI

Mental model: agents are workflows where the edges are decided by an LLM. Most real systems are workflows with a small agentic sub-step (e.g., a research node that's an agent), not end-to-end agents.

Key LangChain v1 Concepts (Memorize)

  • create_agent() — the canonical way to build an agent. Replaces older AgentExecutor and create_react_agent. Handles the tool-call loop for you.
  • @tool — decorator that turns a Python function into an LLM-callable tool. The docstring becomes the tool description; type hints become the schema.
  • checkpointer — persistence layer (e.g., MemorySaver, Postgres). Keyed by thread_id for multi-turn memory.
  • middleware — intercepts the agent loop. Used for human-in-the-loop approval, custom logging, retries, dynamic prompt rewriting.
  • response_format — Pydantic model for structured output; agent returns typed object in result["structured_response"].
  • recursion_limit — pass in invoke config to cap loop iterations (prevents runaway agents).

Example 1 — Agentic System (LangChain v1) for Banking Support

A relationship-manager assistant that can look up products, check rates, draft an email, and require human approval before sending. Demonstrates tools, persistence, structured output, and HITL middleware.

from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langchain_core.tools import tool
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command
from pydantic import BaseModel, Field

# 1. Tools — each is a normal Python function with a clear docstring.
@tool
def lookup_product(product_code: str) -> dict:
    """Look up a banking product by its internal code.

    Args:
        product_code: Internal code, e.g. 'MTG-FIX-5Y' for 5-year fixed mortgage.
    """
    catalogue = {
        "MTG-FIX-5Y": {"name": "5-Year Fixed Mortgage",
                       "rate": 4.29, "erc_schedule": "5,5,3,2,1"},
        "ISA-CASH":   {"name": "Cash ISA",
                       "rate": 4.10, "annual_limit": 20000},
    }
    return catalogue.get(product_code, {"error": "unknown product"})

@tool
def get_client_segment(client_id: str) -> str:
    """Return client segment (mass / affluent / private) for tailoring responses."""
    return "affluent"  # stub — real impl hits CRM

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to a client. THIS IS A WRITE ACTION — requires approval.

    Args:
        to: Recipient email address.
        subject: Email subject line.
        body: Email body in plain text.
    """
    # Real impl: SMTP / Graph API
    return f"sent to {to}"

# 2. Structured output — agent returns a typed response.
class AgentResponse(BaseModel):
    summary: str = Field(description="One-sentence summary for the RM.")
    action_taken: str = Field(description="What the agent did.")
    next_steps: list[str] = Field(default_factory=list)

# 3. Human-in-the-loop on the destructive tool.
hitl = HumanInTheLoopMiddleware(
    interrupt_on={"send_email": True},   # require approval before send
)

# 4. Build the agent.
agent = create_agent(
    model="anthropic:claude-opus-4-7",
    tools=[lookup_product, get_client_segment, send_email],
    system_prompt=(
        "You are a relationship-manager assistant for a retail bank. "
        "Use tools to look up products and client info before answering. "
        "Cite product codes. Never invent rates or fees. "
        "If asked to email a client, draft it and call send_email — "
        "a human will approve before it goes out."
    ),
    checkpointer=MemorySaver(),
    middleware=[hitl],
    response_format=AgentResponse,
)

# 5. Invoke with a thread_id for memory + a recursion limit for safety.
config = {"configurable": {"thread_id": "rm-42-session-1"},
          "recursion_limit": 12}

result = agent.invoke(
    {"messages": [{"role": "user", "content":
        "Client C-9981 is asking about our 5-year fixed mortgage rate. "
        "Look it up and email them a summary at jane@example.com."}]},
    config=config,
)

# 6. The agent pauses at send_email — resume after human approves:
result = agent.invoke(
    Command(resume={"decisions": [{"type": "approve"}]}),
    config=config,
)

print(result["structured_response"].summary)
print(result["structured_response"].next_steps)
What to point out in interview: tool design with clear docstrings, HITL on the destructive action, thread_id for memory, recursion_limit to prevent runaway loops, structured output for downstream systems, and the deliberate choice to not let the agent write to systems of record without approval.

Example 2 — Simple LangGraph Workflow (No Agent): Loan Triage

Linear pipeline with conditional routing. Classify a loan application → route to fast-track, manual-review, or auto-decline. No LLM autonomy — every transition is in code.

from typing import Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END

class LoanState(TypedDict):
    application_id: str
    credit_score: int
    income: float
    requested_amount: float
    decision: str
    rationale: str

def classify(state: LoanState) -> dict:
    score, income, amt = state["credit_score"], state["income"], state["requested_amount"]
    dti = amt / max(income, 1)
    if score >= 740 and dti < 3:
        bucket = "fast_track"
    elif score < 580 or dti > 6:
        bucket = "auto_decline"
    else:
        bucket = "manual_review"
    return {"decision": bucket}

def route(state: LoanState) -> Literal["fast_track", "manual_review", "auto_decline"]:
    return state["decision"]

def fast_track(state: LoanState) -> dict:
    return {"rationale": "Approved on bureau score and low DTI."}

def manual_review(state: LoanState) -> dict:
    return {"rationale": "Routed to underwriter queue for review."}

def auto_decline(state: LoanState) -> dict:
    return {"rationale": "Declined: bureau or DTI thresholds breached."}

graph = (
    StateGraph(LoanState)
    .add_node("classify", classify)
    .add_node("fast_track", fast_track)
    .add_node("manual_review", manual_review)
    .add_node("auto_decline", auto_decline)
    .add_edge(START, "classify")
    .add_conditional_edges("classify", route,
                           ["fast_track", "manual_review", "auto_decline"])
    .add_edge("fast_track", END)
    .add_edge("manual_review", END)
    .add_edge("auto_decline", END)
    .compile()
)

result = graph.invoke({
    "application_id": "APP-1001",
    "credit_score": 760, "income": 90000, "requested_amount": 200000,
    "decision": "", "rationale": "",
})
print(result["decision"], "—", result["rationale"])

Example 3 — LangGraph RAG Workflow (No Agent)

Deterministic RAG pipeline as a graph: retrieve → grade context → generate (or refuse). No model autonomy; you control every step. This is the shape of most production RAG systems.

from typing import Literal, Annotated
import operator
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-opus-4-7", temperature=0)

class RAGState(TypedDict):
    question: str
    contexts: list[str]
    context_ok: bool
    answer: str
    citations: Annotated[list[str], operator.add]

def retrieve(state: RAGState) -> dict:
    chunks = vector_search(state["question"], k=5)  # your retriever
    return {"contexts": [c.text for c in chunks],
            "citations": [c.source for c in chunks]}

def grade_context(state: RAGState) -> dict:
    """LLM judges whether the retrieved context can actually answer the question."""
    judgement = llm.invoke(
        f"Can the question be answered from this context? Reply YES or NO.\n\n"
        f"Q: {state['question']}\nCONTEXT:\n" + "\n---\n".join(state["contexts"])
    ).content.strip().upper()
    return {"context_ok": judgement.startswith("YES")}

def route_after_grade(state: RAGState) -> Literal["generate", "refuse"]:
    return "generate" if state["context_ok"] else "refuse"

def generate(state: RAGState) -> dict:
    ctx = "\n\n".join(f"[{i+1}] {c}" for i, c in enumerate(state["contexts"]))
    msg = llm.invoke(
        f"Answer ONLY from the context. Cite sources as [n].\n\n"
        f"Context:\n{ctx}\n\nQuestion: {state['question']}"
    )
    return {"answer": msg.content}

def refuse(state: RAGState) -> dict:
    return {"answer": "I don't have enough information to answer that confidently."}

graph = (
    StateGraph(RAGState)
    .add_node("retrieve", retrieve)
    .add_node("grade", grade_context)
    .add_node("generate", generate)
    .add_node("refuse", refuse)
    .add_edge(START, "retrieve")
    .add_edge("retrieve", "grade")
    .add_conditional_edges("grade", route_after_grade, ["generate", "refuse"])
    .add_edge("generate", END)
    .add_edge("refuse", END)
    .compile()
)

result = graph.invoke({"question": "What's the ERC on a 5-year fixed mortgage?",
                       "contexts": [], "context_ok": False,
                       "answer": "", "citations": []})
print(result["answer"], "\nSources:", result["citations"])

Example 4 — LangGraph Parallel Fan-Out (No Agent)

Process a batch of client documents in parallel using the Send API. Useful for evaluation pipelines, bulk summarization, or any embarrassingly-parallel work.

from typing import Annotated
import operator
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

class BatchState(TypedDict):
    documents: list[str]
    summaries: Annotated[list[str], operator.add]   # reducer accumulates
    report: str

def fan_out(state: BatchState):
    # Returns a list of Send commands — one worker per document.
    return [Send("summarize", {"doc": d}) for d in state["documents"]]

def summarize(state: dict) -> dict:
    # Each worker runs in parallel.
    return {"summaries": [f"Summary of: {state['doc'][:30]}..."]}

def report(state: BatchState) -> dict:
    return {"report": f"Processed {len(state['summaries'])} documents."}

graph = (
    StateGraph(BatchState)
    .add_node("summarize", summarize)
    .add_node("report", report)
    .add_conditional_edges(START, fan_out, ["summarize"])
    .add_edge("summarize", "report")
    .add_edge("report", END)
    .compile()
)

result = graph.invoke({
    "documents": ["doc A text...", "doc B text...", "doc C text..."],
    "summaries": [], "report": "",
})
print(result["report"])
Q1 When do you choose create_agent vs a hand-built LangGraph workflow?

I default to a workflow because it's predictable, debuggable, and cheaper. I reach for an agent only when the path genuinely can't be enumerated — open-ended research, tool-rich problem solving — and even then I scope it tightly with a small toolset, recursion limits, and HITL on any destructive action. Most production systems I build are workflows with one agentic node, not the other way around.

Q2 What's the role of middleware in LangChain v1?

Middleware sits inside the agent loop and intercepts tool calls or LLM steps. The most important built-in is HumanInTheLoopMiddleware, which pauses execution before specified tools fire and resumes via Command(resume=...) after a human decides. You can also write custom middleware with @wrap_tool_call for logging, redaction, or dynamic prompt rewriting. Middleware is how you add enterprise controls without rewriting the agent.

Q3 What's a reducer in LangGraph and why does it matter?

A reducer defines how concurrent updates to a state field are merged. Default behavior is "last write wins" — fine for scalars but disastrous for lists when nodes run in parallel (Send fan-out). Annotating a list with operator.add makes updates concatenate instead. Forgetting reducers on accumulator fields is the single most common LangGraph bug.

Q4 How do you prevent an agent from looping forever or burning budget?

Three controls: (1) recursion_limit in invoke config — hard cap on loop iterations. (2) Tool design — make tools idempotent, return clear "done" signals, validate args strictly. (3) Token / cost budget per session enforced in middleware — kill the run if it exceeds the cap and surface the partial result. I also log every step to LangSmith so I can replay and tune.
