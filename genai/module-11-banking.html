5c End-to-End Banking Project: "Client Pathways Knowledge Assistant"

A complete project narrative you can walk through in an interview — from raw data to production monitoring. Banking domain. Memorize the shape, swap in your own numbers if you've done a similar project.

The Problem (Frame It in Business Terms)

Relationship managers and contact-center agents at a retail bank handle 40k+ client queries/month spanning products (mortgages, ISAs, credit cards), policies, and onboarding procedures. Average handle time: 8 min, with ~30% of time spent searching across SharePoint, the policy wiki, the product catalogue, and Outlook. Goal: a grounded GenAI assistant that returns cited answers in <3s, cuts handle time by 20%, and never quotes outdated or unauthorized content.

Stage 1 — Data Sourcing & Cleaning

  • Sources — SharePoint (15k policy docs), Confluence (8k pages), Product DB (Postgres), Salesforce KB articles, Outlook FAQs.
  • Connectors — Microsoft Graph API (SharePoint), Confluence REST, JDBC for product DB. Run as scheduled jobs + CDC for high-churn sources.
  • Cleaning steps:
    • PDF → text with unstructured or PyMuPDF; preserve tables via pdfplumber or LayoutLM.
    • Strip headers/footers, page numbers, watermarks ("CONFIDENTIAL — DRAFT").
    • Drop duplicates via MinHash/LSH (banking content is heavily duplicated across regions).
    • Normalize whitespace, fix encoding (smart quotes, ligatures).
    • OCR fallback for scanned policy PDFs (Tesseract or Azure Document Intelligence).
    • Detect language; route non-English to translation pipeline if needed.
  • PII handling — Microsoft Presidio or AWS Comprehend to detect PAN, IBAN, NI numbers, names. Redact if not needed for the use case; tokenize if needed for retrieval.

Stage 2 — Labeling & Metadata Enrichment

  • Why label? Metadata enables filtering ("only mortgage docs valid in 2026"), routing (commercial vs retail), and ACL enforcement.
  • What to label:
    • Document-level: doc_type (policy / FAQ / product sheet), product_line, region, effective_date, expiry_date, sensitivity (Public / Internal / Confidential / Restricted), owner.
    • Chunk-level: section heading, applies_to (customer segment), regulatory_ref (e.g., "FCA CONC 5.2").
  • How:
    • Rule-based extractors for structured fields (filename patterns, doc properties).
    • LLM-assisted labeling for semantic fields (doc_type, applies_to) — sample, verify with SMEs, then scale.
    • SME-in-the-loop for the gold set: 200 docs reviewed by compliance + product SMEs to anchor the labeling rubric and serve as eval ground truth.
    • Tooling: Label Studio or Argilla for the SME workflow; annotators agree on a rubric before labeling at volume.
  • ACL propagation — every chunk inherits the source doc's permission set. Retrieval applies the user's identity as a filter before similarity search.

Stage 3 — Chunking, Embedding & Indexing

  • Chunking — recursive splitter at 800 tokens / 100 overlap, with semantic boundary respect (don't break mid-table, mid-list).
  • Contextual prefix — prepend 1–2 sentences of doc context to each chunk before embedding (Anthropic's Contextual Retrieval). Use prompt caching on the full doc to keep this cheap.
  • Embedding model — start with text-embedding-3-large (1536-d) or voyage-3. Evaluate against domain on the gold set; consider domain-tuned embeddings (FinBERT-style) only if measurable lift.
  • Indexes — pgvector (HNSW index) for dense; OpenSearch for BM25; both keyed on chunk_id with metadata. Reconcile via RRF at query time.
  • Refresh strategy — incremental: CDC events trigger re-embed of changed chunks. Full rebuild quarterly. Track index_version; queries pin a version for reproducibility.

Stage 4 — Retrieval & Generation

  • Query pipeline:
    1. Auth → resolve user identity, roles, regions.
    2. Query rewrite (LLM, fast/cheap model) to expand abbreviations, resolve "this/that" against chat history.
    3. Hybrid retrieval (top-50 dense + top-50 BM25) with hard filters: ACL, region, effective_date <= today <= expiry_date.
    4. RRF fusion → top-30.
    5. Cross-encoder rerank → top-5.
    6. LLM (Claude) with system prompt enforcing: "answer only from context; cite [n]; if unsure, say so."
    7. Post-processor verifies citations exist, redacts any leaked PII, logs full trace.
  • Guardrails — input filter for prompt injection & off-topic; output filter for PII leakage, toxic content, claim-without-citation. Block if violated; log for review.
  • Latency budget — 3s p95: retrieve 200ms, rerank 250ms, LLM streaming first-token 800ms, full response 2s. Stream to UI to mask perceived latency.

Stage 5 — Evaluation (Pre-Production)

  • Gold dataset — 300 (question, ideal_answer, expected_sources) tuples curated with SMEs, covering top intents (rates, eligibility, procedure, exceptions) and edge cases (out-of-scope, multi-hop).
  • Retrieval metrics — Hit@k, MRR, nDCG against expected_sources.
  • Generation metrics — Faithfulness, Answer Relevance, Context Precision/Recall (RAGAS — see next section).
  • Behavioral checks — refusal rate on out-of-scope, citation presence, format compliance.
  • Human review — SMEs blind-rate a 10% sample weekly during development.
  • CI gate — every prompt/model change runs evals; thresholds block merge if any metric regresses >2 points.

Stage 6 — Deployment & Rollout

  • Shadow mode first: real queries go to current process AND to the assistant; compare offline. No user impact.
  • Canary: 5% of agents → 25% → 100% over 3 weeks. Auto-rollback on p95 latency or eval regression.
  • Feature flags for prompt and model versions; A/B testing infrastructure for prompt iteration.
  • Approvals: model risk management (MRM) sign-off, infosec review, data privacy impact assessment (DPIA), legal review of disclaimers.

Stage 7 — Monitoring & Continuous Evaluation

  • System metrics — p50/p95/p99 latency, error rate, tokens/request, $/request, throughput.
  • Quality metrics (online):
    • Implicit: thumbs up/down, copy-to-clipboard rate, escalation-to-human rate, query reformulation rate.
    • LLM-as-judge: sample 5% of live traffic, score on faithfulness + relevance with a judge model. Alert on drift.
    • Citation health: % responses with valid citations, % citations to expired docs.
  • Drift detection — embedding drift (new query distribution), retrieval drift (top-k composition shifting), data drift (new doc types).
  • Feedback loop — thumbs-down + agent comment routed to a review queue; SMEs triage weekly into: prompt fix, doc fix, or new gold-set example.
  • Auditability — every response stored 7 years (regulatory) with prompt version, model version, retrieved chunks, citations, user, timestamp.
  • Cost guardrails — per-user and global token budgets; alerts at 80% of monthly cap.

Outcomes (Have Numbers Ready)

  • Average handle time: 8.0 min → 6.1 min (-24%).
  • Answer accuracy on gold set: 71% → 92% after hybrid + rerank + contextual retrieval.
  • Hallucination rate (faithfulness < 0.8): <1.5% on sampled live traffic.
  • Cost: ~$0.018/query after prompt caching + model routing.
  • Adoption: 78% weekly active among target user pool by month 3.
Interview tip: when asked "tell me about a project," walk this top-to-bottom in 4–5 minutes. Lead with the business problem, not the tech stack. End with a lesson learned (e.g., "the biggest unlock wasn't the model — it was getting compliance to label the corpus correctly").
