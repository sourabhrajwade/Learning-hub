Basics — Dense Vectors, Sparse Vectors & Reranking

Before hybrid search or ColBERT, you need a clear picture of what gets stored in the index and how the two-stage retrieve → rerank pipeline works.

Dense Vectors

A dense vector is a fixed-length array of floats (typically 384–3072 dimensions) where every dimension carries signal. A bi-encoder embedding model (e.g., text-embedding-3-large, bge-m3) reads a chunk of text and outputs one vector that summarizes its meaning. At query time you embed the question the same way and find the nearest vectors by cosine similarity or dot product.

  • Strengths: semantic matching — "5-year fixed mortgage" matches "five-year fixed-rate home loan" even with no shared keywords.
  • Weaknesses: misses exact IDs, product codes, acronyms, rare proper nouns. One vector per chunk = coarse representation.
  • Index: stored in a vector DB or pgvector; searched with ANN algorithms (HNSW, IVF) for speed at scale.
  • Typical dims: 384 (small models), 768 (BERT-era), 1024 (BGE-large), 1536 (OpenAI), 3072 (OpenAI large).

Sparse Vectors

A sparse vector is mostly zeros — only a few dimensions are non-zero, each tied to a vocabulary term (or learned feature). Classic keyword search (BM25, TF-IDF) is sparse retrieval without neural embeddings: "mortgage" and "ISA" hit specific term dimensions.

  • BM25 — inverted-index keyword ranking. Fast, explainable, great for SKUs, policy numbers, legal citations.
  • Learned sparse (SPLADE, uniCOIL) — a transformer expands text into a sparse weight vector over the vocabulary. Gets semantic expansion ("car" → also weights "vehicle") while staying sparse and fast on inverted indexes.
  • Why hybrid? Dense and sparse fail in opposite ways. Fuse both with Reciprocal Rank Fusion (RRF) or weighted score blending.

Reranking — The Two-Stage Pattern

Stage 1 (retrieval): cast a wide net — top-50 or top-100 candidates via fast bi-encoder similarity, BM25, or hybrid. Stage 2 (reranking): a slower, more accurate model scores each (query, document) pair and reorders to top-5 for the LLM.

  • Bi-encoder — embed query and doc separately; similarity = dot product. O(n) over the corpus. Used for first pass.
  • Cross-encoder reranker — feed query + doc together through one transformer; outputs a single relevance score. O(k) over candidates only. Much more accurate.
  • Cost trade-off: reranking 50 docs adds ~100–300ms but is often the single biggest quality jump after hybrid search.
  • Examples: Cohere Rerank v3, bge-reranker-v2-m3, Voyage rerank, ms-marco-MiniLM-L-6-v2 (small/fast).

Intermediate — ColBERT, MRL & Hybrid Retrieval

Once basics work, these techniques improve recall and efficiency without jumping to a full custom stack.

ColBERT — Late Interaction Retrieval

ColBERT (Contextualized Late Interaction over BERT) sits between bi-encoders and cross-encoders. Instead of one vector per document, it stores a multi-vector representation — one embedding per token. At query time, each query token finds its best-matching doc token; scores are aggregated via MaxSim (sum of max similarities per query token).

  • Why it helps: finer-grained matching than a single pooled vector, but still indexable (unlike a full cross-encoder over the whole corpus).
  • Variants: ColBERTv2 (compressed token embeddings), Jina-ColBERT, ColQwen (multimodal pages).
  • Trade-off: more storage per doc (many vectors vs one), more complex indexing. Often used as a reranker or on smaller corpora.
  • Interview line: "Bi-encoder for speed, ColBERT for late interaction, cross-encoder for final rerank."

MRL — Matryoshka Representation Learning

MRL trains embeddings so the first N dimensions of a vector are still meaningful when you truncate. A 3072-d embedding from text-embedding-3-large can be cut to 1536 or 512 dims with modest quality loss — like Russian nesting dolls where inner layers still work.

  • Storage savings: store 512-d instead of 3072-d → ~6× less disk and RAM.
  • Latency: smaller vectors = faster distance computation and smaller HNSW graphs.
  • Multi-resolution: same model, different dims per use case — 1536 for quality-critical, 256 for high-volume dedup.
  • Models with MRL: OpenAI embedding-3 family, Nomic Embed, some BGE variants. Always validate truncation on your eval set.

Advanced — Qdrant, Learned Sparse, Small Models & Production Tuning

Production-grade retrieval stacks where vector DB choice, model size, and index configuration matter as much as algorithm choice.

Qdrant — Vector DB Deep-Dive

Qdrant is a Rust-native vector database built for filtered hybrid search at scale. Strong choice when you outgrow pgvector or need native sparse + dense in one engine.

  • Collections & points — a collection holds points; each point has an ID, one or more named vectors, and a JSON payload (metadata).
  • Named vectors — store dense (1536-d) and sparse (SPLADE) on the same point; query both in one request.
  • Payload filtering — apply ACL/region/date filters inside the HNSW search (pre-filter or post-filter). Critical for banking ACL enforcement.
  • HNSW tuning — m (edges per node), ef_construct (build quality), ef (search recall). Higher = better recall, more RAM and slower queries.
  • Quantization — scalar or product quantization compresses vectors (e.g., 4× RAM reduction) with configurable accuracy trade-off.
  • Multitenancy — shard by tenant; use payload indexes on hot filter fields (tenant_id, region).
  • When to pick Qdrant over pgvector: >10M vectors, need hybrid sparse+dense natively, heavy metadata filtering, dedicated ops team. Stay on pgvector if you're <5M docs and already on Postgres.

Learned Sparse Vectors (SPLADE) — Advanced Hybrid

SPLADE (and successors like SPLADE-v3, uniCOIL) train a transformer to output sparse term weights. You get BM25-like inverted-index speed with neural term expansion. Qdrant, Elasticsearch, and OpenSearch all support sparse vector fields. Pair with dense in RRF for state-of-the-art hybrid on BEIR-style benchmarks.

ColBERT in Production

  • Indexing: store token embeddings per chunk (often 100–300 vectors × 128-d each after compression in ColBERTv2).
  • Retrieval: MaxSim scoring — for each query token, find max dot-product with any doc token, sum across query tokens.
  • Placement: use as a second-stage retriever after dense top-200, or as a reranker before cross-encoder. Rarely the only retrieval method at >1M docs without dedicated infra (RAGatouille, Vespa).
  • Multimodal: ColPali / ColQwen apply the same late-interaction idea to page images — strong for layout-heavy PDFs.

Small Models — Embedding & Reranking at the Edge

Not every stage needs a frontier model. Small models (sub-500M params) run on CPU or a single GPU, cut latency and cost, and work well for high-QPS retrieval tiers.

MRL in Production — Practical Checklist

  1. Embed at full dimension (e.g., 1536) during indexing.
  2. Run recall@k eval at 1536, 768, 512, 256 — find the knee of the curve.
  3. Re-index at chosen dim OR truncate at query time if model supports MRL.
  4. Re-tune HNSW ef — lower dims sometimes need higher ef to compensate.
  5. Document the chosen dim in index_version metadata for reproducibility.
Real-world stack you'll most often deploy: Hybrid search (BM25 + dense) → metadata filter → top-50 → cross-encoder rerank → top-5 → LLM with citations. That alone beats 90% of "naive RAG" implementations.
Q1 Explain Contextual Retrieval and why it works.

Standard chunking embeds each chunk in isolation — but a chunk like "the rate is 4.5%" is meaningless without knowing it's about a 30-year fixed mortgage in the 2024 product guide. Contextual Retrieval uses an LLM (cheaply, with prompt caching on the full doc) to generate a short situating context for each chunk and prepends it before embedding and BM25 indexing. The chunk now retrieves correctly because its embedding reflects its true meaning. Anthropic reported ~49% reduction in retrieval failures combined with reranking.

Q2 When would you use GraphRAG over vector RAG?

GraphRAG shines on global, multi-hop questions — "summarize the top risks across all 200 audit reports" or "how is entity X connected to entity Y across these documents?" Vector RAG retrieves local semantic neighbors and struggles with these. GraphRAG extracts entities/relations into a graph, clusters into communities, and pre-summarizes each community. Cost is much higher up front (LLM calls per document for extraction), so use it when query patterns justify it.

Q3 What's reranking and why is it worth the latency?

Embedding similarity is a bi-encoder — query and doc embedded independently. Fast but loses cross-attention nuance. A cross-encoder reranker takes (query, doc) pairs together and outputs a relevance score, capturing fine-grained interaction. Pattern: retrieve top-50 fast (bi-encoder), rerank to top-5 (cross-encoder). Adds 100–300ms but often the single biggest quality jump in the pipeline.

Q4 What's HyDE and when does it help?

HyDE (Hypothetical Document Embeddings): the LLM generates a hypothetical answer to the question, then you embed and retrieve based on that synthetic answer rather than the original question. Helps when the question and the document corpus use very different language — short questions vs. long technical docs. Adds an LLM call to the critical path, so weigh latency.

Q5 Dense vs sparse vectors — when do you need both?

Dense vectors capture semantic meaning — paraphrases, synonyms, conceptual similarity. Sparse vectors (BM25 or learned SPLADE) capture lexical overlap — product codes, legal references, acronyms, exact names. In banking RAG you almost always need both: dense finds "early repayment charge" when the user says "pay off mortgage early fee"; sparse finds "SKU-88421" or "FCA CONC 5.2.1R". Fuse with RRF; don't rely on dense alone.

Q6 Explain ColBERT and how it differs from bi-encoder and cross-encoder.

A bi-encoder pools the whole document into one vector — fast but coarse. A cross-encoder runs query and doc through shared attention — accurate but too slow for millions of docs. ColBERT stores one vector per token in the document and scores via MaxSim: each query token matches its best doc token, scores are summed. You get token-level interaction without scoring every (query, doc) pair at index time. Typical pattern: bi-encoder top-200 → ColBERT rescore → cross-encoder rerank top-5.

Q7 What is MRL and why would you truncate embedding dimensions?

Matryoshka Representation Learning (MRL) trains embeddings so lower-dimensional prefixes remain useful. Truncating 1536-d → 512-d cuts storage and speeds up HNSW search with ~5–15% recall loss (model-dependent). Use it when you have millions of vectors and RAM/disk is the bottleneck, or when you want one model serving both a high-quality tier (full dims) and a fast dedup tier (256-d). Always measure on your gold set — don't truncate blindly.

Q8 Why Qdrant over pgvector or Pinecone?

pgvector wins when you're <5–10M vectors, already on Postgres, and want one database. Qdrant wins when you need native hybrid sparse+dense on the same point, heavy payload filtering inside HNSW, quantization at scale, or >10M vectors with dedicated indexing. Pinecone wins when you want fully managed with minimal ops. For regulated banking: Qdrant self-hosted or Qdrant Cloud with private networking gives you ACL payload filters and audit-friendly point-level metadata.

Q9 When do small embedding/reranking models beat frontier APIs?

Small models (bge-small, all-MiniLM, ms-marco-MiniLM reranker) win on cost, latency, and data residency — self-host on CPU, no per-token bill, data never leaves your VPC. They lose on multilingual coverage, domain jargon, and absolute recall. Rule: start with a small open model + your eval set; if recall@10 on the gold set is >85%, ship it. If not, step up to bge-m3 or voyage-3 for embeddings and bge-reranker-v2 or Cohere Rerank for the rerank stage. Use small LLMs (Phi-3-mini, Haiku) only for routing/rewrite — not final regulated answers.
