Module 6 · Scalability for Large Apps Advanced

6.1 Monorepos

One repo, many packages. Tools: Nx, Turborepo, pnpm workspaces, Bazel, Rush.

  • Shared tooling, atomic cross-package changes, single-version policy.
  • Affected-only builds (Nx affected, Turbo task graph) = fast CI.
  • Remote caching across the team.
  • Trade-off: tighter coupling, complex CI, larger repo. Worth it past ~3-5 collaborating apps.

6.2 Design Systems

  • Tokens (color, spacing, typography) generated to CSS vars / native platforms.
  • Primitives (Button, Input, Modal) — accessible, themable.
  • Patterns (Form, Table, EmptyState) — composed primitives.
  • Docs with Storybook, automated visual regression tests (Chromatic, Percy).
  • Versioned releases; deprecation policy; migration codemods.

6.3 Micro-Frontends

Independent teams ship independent frontends, composed at runtime or build time. Useful when team count > ~5 and release cadences differ.

Composition strategies

  • Build-time: npm packages composed in a shell. Simplest; coupled releases.
  • Server-side composition: shell stitches HTML fragments (Tailor, Podium, Edge Includes).
  • Run-time / Module Federation (Webpack 5): host loads remotes at runtime, sharing React/etc.
  • Iframes: strong isolation; bad for UX (focus, perf, deep links).
  • Web Components: framework-agnostic boundary.
Costs: dependency duplication, version skew, design inconsistency, perf regressions, cross-MFE communication complexity. Don't adopt for small teams.

6.4 Internationalization & Accessibility

i18n

  • ICU MessageFormat for plurals/genders; tools: react-intl, i18next, FormatJS.
  • RTL support: logical properties (margin-inline-start), dir="rtl".
  • Locale-aware sorting, dates (Intl.DateTimeFormat), numbers, currency.
  • Lazy-load locale bundles by route.

a11y

  • Semantic HTML first; ARIA only when needed.
  • Keyboard support: tab order, focus management, skip links.
  • Color contrast (WCAG AA: 4.5:1 text), prefers-reduced-motion.
  • Automated tests: axe-core, Lighthouse; manual: screen reader passes.

6.5 Observability & Releases

  • RUM (Real User Monitoring): web-vitals lib → analytics endpoint.
  • Error tracking: Sentry, Bugsnag — source maps for stack traces.
  • Feature flags: LaunchDarkly, Statsig; gradual rollouts, kill switches.
  • Releases: canary deploys, automatic rollback on regressed CWV/error rate.
  • Synthetic monitoring: Lighthouse CI, Playwright on schedule.
Q: 5 teams want to ship to one app. Do you go micro-frontend?
Not automatically. First try: monorepo + clear module ownership + design system + trunk-based development with feature flags. MFEs only if teams need independent deploys, different tech stacks, or have hard organizational boundaries (e.g., acquisitions). The cost of MFE is real: duplicate deps, runtime contracts, perf overhead.
