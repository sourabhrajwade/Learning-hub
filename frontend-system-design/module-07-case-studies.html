7.1 Twitter / News Feed

Requirements: Infinite-scroll timeline, real-time updates, like/retweet, media, < 1s perceived load.

Key decisions

  • Pagination: cursor-based (?since_id=...&max_id=...). Cursors are stable under inserts; offset isn't.
  • Virtualization for long feeds + dynamic heights (TanStack Virtual).
  • Optimistic updates for like/retweet.
  • WebSocket / SSE for "X new tweets" pill — don't auto-prepend; UX problem.
  • Image strategy: low-quality placeholder (LQIP) → progressive enhancement; fetchpriority="high" for above-the-fold.
  • Cache normalization: store tweets by ID once; lists store IDs only — single update propagates everywhere.

Data model (client cache)

state = {
  tweets: { [id]: TweetEntity },
  feeds: {
    home:    { ids: [...], cursor: '...', loading: false },
    profile_42: { ids: [...], cursor: '...' }
  }
}

7.2 Chat (WhatsApp / Slack)

  • Transport: WebSocket primary, fallback to long-polling. Heartbeats for liveness.
  • Offline-first: IndexedDB store; outbox queue for unsent messages with exponential retry.
  • Ordering & dedup: client-generated IDs (UUID/ULID) replaced by server IDs on ack — keys stay stable.
  • Read receipts & typing: debounced events; coalesce server-side.
  • Reconnection: on resume, send last-known seq number; server replays missed.
  • Scroll UX: virtualization, "scroll to bottom" only when user is near bottom; stickiness.
  • E2E encryption (WhatsApp): keys never leave client; web client mirrors mobile via QR pairing.

7.3 Google Docs (Real-time Collaboration)

Core problem: two users edit simultaneously — how do edits converge?

  • OT (Operational Transform): server transforms ops against concurrent ops; used by Google Docs.
  • CRDTs (Yjs, Automerge): operations are commutative; works peer-to-peer; popular in newer tools.
  • Presence: awareness protocol with cursors and selections.
  • Offline editing: CRDTs shine — merge on reconnect.
  • Persistence: snapshot + op log; periodic compaction.
  • Conflict-free formatting: formatting as ranges with stable identifiers.

7.4 Netflix (Video Streaming UI)

  • Adaptive Bitrate Streaming (HLS / DASH): MPD/manifest lists segments at multiple bitrates; client picks based on bandwidth + buffer.
  • Pre-fetching: next episode metadata + thumbnails; key art preloaded for hovered tile.
  • Skeleton + low-bitrate poster while metadata loads.
  • Personalization at edge — cacheable shell + per-user rows JSON.
  • Memory management: long sessions; lazy unmount off-screen rows; revoke blob: URLs.
  • DRM: EME/Widevine/FairPlay/PlayReady — license server + content keys.
  • Subtitles: WebVTT, accessibility considerations.

7.5 Autocomplete / Typeahead

The classic frontend system design question.

  • Debounce input (~150-300ms) so we don't fire per keystroke.
  • Cancel in-flight requests on new keystroke (AbortController).
  • Race conditions: drop responses for stale queries — track latest query id.
  • Caching: per-prefix LRU; "abc" results contain "abcd" candidates often.
  • Keyboard nav: ↑/↓/Enter/Esc; ARIA role="combobox", aria-activedescendant.
  • Trie/server side when offline-first or low latency required.
const ctrl = useRef<AbortController>();
async function search(q: string) {
  ctrl.current?.abort();
  ctrl.current = new AbortController();
  const data = await fetch(`/search?q=${encodeURIComponent(q)}`,
    { signal: ctrl.current.signal }).then(r => r.json());
  setResults(data);
}

7.6 E-commerce (Amazon-style)

  • Listing pages: SSR/ISR for SEO, faceted filtering, pagination + URL state.
  • Product detail: cached HTML shell + dynamic price/stock fragment (edge personalization).
  • Cart: client + server source of truth; merge on login.
  • Checkout: isolated bundle, strict CSP, idempotency keys for order submit, no third-party scripts beyond payment.
  • Image CDN: per-device responsive variants, AVIF/WebP, lazy below-the-fold.
  • Search: typeahead → results page; query state lives in URL.
