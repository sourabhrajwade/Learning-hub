2 Generative AI Solutions

Expect end-to-end thinking: use case selection, RAG, prompt engineering, evaluation, guardrails, cost.

Q1 How do you decide whether a problem is a good fit for GenAI?

I screen on four axes: (1) language-shaped — does the input/output involve unstructured text, code, or images? (2) tolerance for non-determinism — can the workflow accept "good most of the time" with a human in the loop? (3) data availability — do we have the right context (docs, examples, tools) to ground the model? (4) ROI vs simpler alternatives — would a rules engine, classifier, or search solve it for 10% the cost? If three of four are "yes," GenAI is on the table. Otherwise, push back.

Q2 Walk me through a RAG architecture you'd build for a "Client Pathways" knowledge assistant.

Ingest: connectors to source systems (CMS, policy docs, CRM notes), chunking with semantic boundaries, enrichment with metadata (product line, region, effective date). Index: hybrid retrieval — BM25 + dense embeddings — in a vector DB (pgvector, Pinecone, or OpenSearch). Retrieve: query rewriting, metadata filtering, reranking with a cross-encoder. Generate: prompt with citations enforced, structured output where possible. Evaluate: faithfulness, context recall, answer relevance via LLM-as-judge plus a hand-labeled gold set. Govern: per-document ACLs propagated to retrieval; PII redaction at ingest.

Q3 What's the difference between fine-tuning, RAG, and prompt engineering — and how do you choose?

Prompt engineering shapes behavior with no weight changes — fastest, cheapest, first thing to try. RAG injects fresh, authoritative context at inference — best for knowledge that changes or is private. Fine-tuning changes weights — best for style, format compliance, or a narrow task on which prompts can't get to quality. Order: prompt → RAG → fine-tune. Fine-tuning is the last lever, not the first, because of cost, drift, and the need for a labeled dataset.

Q4 How do you evaluate a GenAI system rigorously?

I build an eval harness early — before scaling. Datasets: a hand-curated golden set (50–200 cases) plus a regression set grown from real traffic. Metrics: task-specific (exact match, ROUGE for summaries), RAG-specific (faithfulness, context precision/recall), behavioral (refusal rate, format compliance). Judges: deterministic checks first; LLM-as-judge with calibrated rubrics; human spot-check on a sample. CI integration: evals run on every prompt or model change, with thresholds that block regressions.

Q5 How do you control cost and latency in a production LLM app?

Cost: prompt caching for stable system prompts (huge win on Anthropic), model routing — cheap model for easy cases, escalate on uncertainty, output token limits, summarize long histories instead of resending. Latency: stream responses, parallelize independent calls, cache deterministic responses, keep retrieval under a tight budget. Observability: p50/p95/p99 latency and tokens-per-request dashboards tied to user-visible KPIs.

Tell a story: have one concrete GenAI project ready with numbers — "we cut handle time by X%, saved Y FTE-hours/month, with a Z% accuracy on the eval set."
