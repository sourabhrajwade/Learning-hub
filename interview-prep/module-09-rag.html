5a RAG Deep-Dive (Study Notes + Interview Q&A)

RAG = Retrieval-Augmented Generation. The model is given fresh, authoritative context at query time so it can answer questions about data it was never trained on. This is the single most common GenAI pattern in enterprise.

Study Notes — The 5 Stages of RAG

  • 1. Ingestion — pull documents from sources (SharePoint, Confluence, S3, DBs). Strip noise (headers, footers, boilerplate). Convert PDFs/HTML/DOCX to clean text. Attach metadata (source, author, date, ACL).
  • 2. Chunking — split documents into retrievable units. Too small → loss of context. Too large → noisy retrieval and wasted tokens. Typical: 500–1000 tokens with 10–20% overlap. Prefer semantic chunking (split on headings/paragraphs) over fixed-size.
  • 3. Embedding & Indexing — convert each chunk into a dense vector (e.g., text-embedding-3-large, voyage-3, bge-m3) and store in a vector DB (pgvector, Pinecone, Weaviate, Qdrant, OpenSearch). Index metadata for filtering.
  • 4. Retrieval — embed the query, find top-k similar chunks (k=4–10 typical). Apply metadata filters (e.g., only docs the user can see). Hybrid retrieval = dense (semantic) + sparse (BM25 keyword) merged with Reciprocal Rank Fusion (RRF).
  • 5. Generation — stuff retrieved chunks into the prompt as context, instruct the LLM to answer using only that context, and to cite sources. Return answer + citations.

Key Concepts to Memorize

TermPlain English
EmbeddingA list of numbers (e.g., 1536 floats) that represents the meaning of a piece of text. Similar meaning → similar vectors.
Cosine similarityAngle between two vectors. 1.0 = identical meaning, 0 = unrelated. Default similarity metric for retrieval.
BM25Classic keyword-based ranking (TF-IDF on steroids). Catches exact-match terms that embeddings miss (e.g., product codes, IDs).
Top-kHow many chunks you retrieve. Higher k = more context but more noise & cost.
RerankerA second model (cross-encoder) that re-scores the top-k for relevance. Slower but much more accurate than pure embedding similarity.
Dense vectorFixed-length float array (e.g., 1536-d) from a bi-encoder; every dimension carries signal. Used for semantic ANN search.
Sparse vectorMostly-zero vector over vocabulary terms (BM25, SPLADE). Catches exact keywords and learned term expansions.
ColBERTLate-interaction retriever: one vector per token, scored via MaxSim. Between bi-encoder speed and cross-encoder accuracy.
MRLMatryoshka embeddings — truncate to fewer dimensions (e.g., 1536→512) with modest quality loss; saves storage and latency.
Chunk overlapRepeating the last N tokens of one chunk at the start of the next, so context isn't lost at boundaries.
GroundingThe answer is derived from the retrieved context, not the model's parametric memory.
FaithfulnessDoes every claim in the answer come from the retrieved context? (Anti-hallucination metric.)
Q1 Why use RAG instead of just fine-tuning the model on your data?

RAG separates knowledge from reasoning. Fine-tuning bakes facts into weights — they go stale, are hard to update, can't enforce ACLs, and can't be cited. RAG keeps facts in a queryable store: update a doc and the next query sees it; remove a doc and the model forgets it. You get freshness, attributability, and access control for free. Fine-tune for style and format; RAG for knowledge.

Q2 Walk me through chunking strategies and how to choose.

Fixed-size (e.g., 800 tokens, 100 overlap) — simple, fast, but cuts mid-sentence. Recursive — splits on natural separators (paragraph → sentence → word) to stay under a size cap; good default. Semantic — splits where embedding similarity drops; expensive but high quality for heterogeneous docs. Document-aware — split by headings for Markdown, by rows for tables, by function for code. Rule of thumb: start recursive @ 800/100, measure retrieval quality, only get fancier if metrics demand it.

Q3 Naive RAG often fails — what are the top failure modes?

(1) Bad retrieval — the right chunk isn't in top-k. Caused by poor chunking, weak embeddings, or query/doc vocabulary mismatch. (2) Lost in the middle — LLMs ignore context buried in long prompts. (3) Hallucination despite context — model answers from training data instead of provided context. (4) No-answer cases — model fabricates rather than saying "I don't know." (5) Stale index — docs updated but vectors not re-computed. (6) ACL leaks — user sees content they shouldn't because filters weren't applied.

Q4 Code: a minimal RAG pipeline.
from anthropic import Anthropic
import psycopg2  # with pgvector extension

client = Anthropic()

def retrieve(query: str, k: int = 5):
    q_vec = embed(query)  # your embedding fn
    rows = db.execute(
        """SELECT content, source FROM chunks
           WHERE acl_check(user_id, doc_id)
           ORDER BY embedding <=> %s LIMIT %s""",
        (q_vec, k),
    ).fetchall()
    return rows

def answer(query: str, user_id: str) -> str:
    chunks = retrieve(query)
    context = "\n\n".join(f"[{i+1}] {c}" for i, (c, _) in enumerate(chunks))
    sources = [s for _, s in chunks]

    msg = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=800,
        system=("Answer ONLY from the context. Cite sources as [1], [2]. "
                "If the context doesn't contain the answer, say 'I don't know.'"),
        messages=[{"role": "user",
                   "content": f"Context:\n{context}\n\nQuestion: {query}"}],
    )
    return msg.content[0].text, sources
