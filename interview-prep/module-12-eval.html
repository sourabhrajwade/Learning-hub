Evaluation frameworks, metrics, and production quality gates.

5d Evaluation: With and Without RAGAS

Evaluation is what separates a demo from a production system. Know the metrics by name, what they measure, and how to compute them with and without a framework.

Study Notes — The Two Halves of RAG Eval

  • Retrieval eval — did we get the right chunks? (Independent of the LLM.)
  • Generation eval — given those chunks, is the answer good?

Always measure both — a great answer with bad retrieval is luck; bad answer with great retrieval is a prompt/model problem.

Core Metrics (Definitions)

MetricWhat it measuresHow
Hit@kWas at least one expected source in top-k?Compare retrieved IDs vs ground-truth IDs.
MRRMean Reciprocal Rank — how high was the first correct hit?1/rank of first relevant doc, averaged.
nDCG@kQuality of ranking, with graded relevance and position discountStandard IR metric; needs graded labels.
FaithfulnessAre all claims in the answer supported by the context?Decompose answer into claims; for each, check entailment from context. (LLM-judged.)
Answer RelevanceDoes the answer actually address the question?LLM generates N questions from the answer; measure cosine similarity to original question.
Context PrecisionOf the retrieved chunks, how many are actually relevant? (Are we sending noise to the LLM?)Per chunk, LLM-judge "is this relevant to the question?" → ranked precision.
Context RecallOf the info needed, how much was retrieved? (Did we miss something?)Decompose ground-truth answer into claims; check each is supported by retrieved context.
Answer CorrectnessDoes the answer match the ground-truth answer?Combination of factual overlap + semantic similarity.

What is RAGAS?

RAGAS (Retrieval-Augmented Generation Assessment) is an open-source Python framework that implements the metrics above out of the box. It uses an LLM as a judge for faithfulness, relevance, context precision/recall, and answer correctness. Reference-free where possible (faithfulness needs no ground truth) — reference-based where required (context recall and answer correctness need ground-truth answers).

Why use it: standard, comparable, fast to set up, integrates with LangChain/LlamaIndex.
Why not (or augment): LLM-judge introduces its own bias; metrics are aggregate, not always diagnostic; you may need domain-specific metrics it doesn't ship.

Evaluating WITH RAGAS — Code

from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall, answer_correctness,
)

# Each row: a question + your system's outputs + (optional) ground truth
data = Dataset.from_list([
    {
        "question": "What's the early-repayment charge on a 5-year fixed mortgage?",
        "answer": "For a 5-year fixed, ERC is 5% in years 1–2, tapering to 1% in year 5 [1].",
        "contexts": [
            "Product Guide 2026 §4.2: ERC schedule for 5-year fixed: Y1 5%, Y2 5%, Y3 3%, Y4 2%, Y5 1%.",
            "FAQ: Customers may overpay 10% per year without ERC.",
        ],
        "ground_truth": "ERC starts at 5% and reduces to 1% by year 5.",
    },
    # ... 200+ more rows
])

results = evaluate(
    data,
    metrics=[faithfulness, answer_relevancy,
             context_precision, context_recall, answer_correctness],
)

print(results)
# {'faithfulness': 0.94, 'answer_relevancy': 0.91,
#  'context_precision': 0.83, 'context_recall': 0.88,
#  'answer_correctness': 0.86}

Run on every prompt/model change. Threshold per metric in CI; failure blocks merge.

Evaluating WITHOUT RAGAS — Roll-Your-Own

When you can't use RAGAS (regulated environment, custom domain metrics, judge-model bias concerns), compute the same ideas yourself. It's not magic — it's structured LLM judging plus deterministic checks.

from anthropic import Anthropic
import json

judge = Anthropic()

def faithfulness_score(answer: str, contexts: list[str]) -> float:
    """Decompose answer into atomic claims; check each is supported."""
    claims = decompose_into_claims(answer, judge)  # LLM call
    supported = 0
    joined_ctx = "\n\n".join(contexts)
    for claim in claims:
        msg = judge.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=10,
            system="Reply only YES or NO.",
            messages=[{"role": "user", "content":
                f"Is this claim supported by the context?\n\n"
                f"CONTEXT:\n{joined_ctx}\n\nCLAIM: {claim}"}],
        )
        if msg.content[0].text.strip().upper().startswith("YES"):
            supported += 1
    return supported / max(len(claims), 1)

def hit_at_k(retrieved_ids: list[str], expected_ids: list[str], k: int = 5) -> int:
    return int(any(rid in expected_ids for rid in retrieved_ids[:k]))

def mrr(retrieved_ids: list[str], expected_ids: list[str]) -> float:
    for i, rid in enumerate(retrieved_ids, 1):
        if rid in expected_ids:
            return 1.0 / i
    return 0.0

def citation_validity(answer: str, allowed_sources: set[str]) -> bool:
    cited = extract_citations(answer)  # regex on [n] markers → source IDs
    return all(c in allowed_sources for c in cited)

# Aggregate over the gold set
def run_eval(gold_set, system):
    results = []
    for row in gold_set:
        out = system(row["question"], user_id=row["user_id"])
        results.append({
            "hit@5": hit_at_k(out.retrieved_ids, row["expected_ids"]),
            "mrr": mrr(out.retrieved_ids, row["expected_ids"]),
            "faithfulness": faithfulness_score(out.answer, out.contexts),
            "citations_ok": citation_validity(out.answer, set(out.retrieved_ids)),
        })
    return aggregate(results)

Domain-specific metrics you should add for banking: regulatory citation accuracy (does it cite the correct FCA/PRA reference?), numerical accuracy (rates, dates, fees parsed and matched exactly), refusal correctness (model refuses on out-of-scope as it should).

Q1 Walk me through how you evaluate a RAG system end-to-end.

Two halves: retrieval and generation. Retrieval — Hit@k, MRR, nDCG against a gold set of (question, expected source IDs). Generation — Faithfulness, Answer Relevance, Context Precision/Recall using RAGAS or my own LLM-judge harness. I run these in CI on every prompt/model change with blocking thresholds. In production, I sample 5% of live traffic for online faithfulness scoring and watch for drift, plus implicit signals like thumbs and reformulation rates.

Q2 What are RAGAS's limitations and how do you compensate?

Judge-model bias — same model family judging itself inflates scores. I cross-check with a different judge (e.g., Claude judges GPT, GPT judges Claude). Aggregate scores hide failures — I slice by intent, doc type, user segment to find weak spots. Doesn't capture domain truth — for banking I add deterministic checks (numerical exact-match on rates and dates, regulatory citation correctness) that no general framework can give. Cost — LLM-as-judge is expensive at scale, so I sample.

Q3 How do you build a gold/eval dataset?

Seed with real production queries (anonymized) — 200–500 covering the intent distribution. Curate ideal answers and expected sources with SMEs. Stratify across intents, doc types, and difficulty (easy / multi-hop / out-of-scope). Grow the set monthly with regression cases from production failures. Version it in git. Hold out a sealed set the team can't see during prompt iteration to prevent overfitting.

Q4 How do you monitor a deployed RAG system?

System health: latency p50/p95/p99, error rate, tokens, cost. Quality online: sample-based LLM-judge faithfulness and relevance, citation validity rate, refusal-rate on out-of-scope. Implicit feedback: thumbs, copy rate, escalation rate, query reformulation rate. Drift: embedding distribution shift on incoming queries, top-k composition shift, new doc-type distribution. Alerts on threshold breaches, with feedback feeding the gold set and doc remediation queue weekly.

Interview soundbite: "I treat eval as a first-class product surface — you can't ship what you can't measure, and you can't improve a RAG system on vibes."
