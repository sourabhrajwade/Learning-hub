L2 GenAI Foundations: How LLMs Actually Work

Before talking about RAG or agents, you need a working mental model of what an LLM is doing. This section explains tokens, embeddings, attention, context windows, sampling, and why "hallucination" is structural — not a bug.

Tokens — The Atom of LLMs

LLMs don't see words or characters — they see tokens. A token is a chunk of text, usually 3–4 characters. "Hello world" might be 2 tokens; "incomprehensibility" might be 4. The vocabulary is fixed (~100k tokens for modern models).

Why it matters: you pay per token, context windows are sized in tokens, and rare words / non-English / code use more tokens than you'd expect. A rule of thumb: 1 token ≈ 0.75 words in English.

Embeddings — Meaning as Geometry

Every token (and every chunk of text) gets converted into a list of numbers — a vector, typically 768 to 4096 dimensions. The clever bit: similar meanings end up near each other in this space.

  • "king" and "queen" are close.
  • "king" - "man" + "woman" ≈ "queen" (the famous example).
  • "5-year fixed mortgage" and "five-year fixed-rate home loan" are close even though they share few words.

This is what makes semantic search work — and it's the foundation of RAG. We're not matching keywords; we're matching meaning.

The Transformer & Attention (Plain English)

A transformer is a neural network that processes a sequence of tokens. Its key innovation is self-attention: at every layer, every token computes how relevant every other token is to it, and combines information accordingly.

Imagine reading the sentence "The bank approved the loan because it was creditworthy." When you encounter "it," your brain instantly looks back to figure out what "it" refers to. Attention does the same thing mathematically — and across all tokens at once, in parallel. That parallelism is why transformers train so fast on GPUs.

What "Generation" Actually Is

An LLM does one thing: given the tokens so far, predict a probability distribution over the next token. To generate a paragraph, it does this thousands of times in a loop, feeding each chosen token back as input.

  • Greedy decoding — always pick the highest-probability token. Deterministic but boring.
  • Temperature — knob that controls randomness. 0 = greedy. 1 = use the raw probabilities. >1 = more random/creative. For factual banking work: temperature 0 or close to it.
  • Top-p (nucleus) — sample only from the smallest set of tokens whose probabilities sum to p (e.g., 0.9). Cuts off the long tail of unlikely tokens.
  • Top-k — sample from only the top k most likely tokens.

Crucial implication: the model has no internal "is this true?" check. It's predicting plausible next tokens. Plausibility ≠ truth. That's why hallucination is structural, not a bug — and why RAG and grounding matter so much.

Context Window — The LLM's Working Memory

The context window is how many tokens the model can "see" at once: system prompt + chat history + retrieved docs + the new user message + the response so far. Modern models: 200k tokens (Claude), 1M (Gemini), 128k (GPT-4).

  • The model has no memory beyond the context window — every API call is stateless.
  • "Conversation memory" is just resending prior turns each call.
  • Cost and latency both scale with context length — bigger isn't always better.
  • Lost-in-the-middle: models pay most attention to the start and end of context. Bury a critical fact at token 50,000 and the model may ignore it.

How LLMs Are Trained (3 Stages)

  1. Pre-training — predict the next token across trillions of tokens of internet text. This is where almost all the world knowledge comes from. Costs $10M–$100M+ in compute. Result: a "base model" that completes text but isn't yet helpful.
  2. Supervised fine-tuning (SFT) — show the model thousands of high-quality (instruction, response) pairs to teach it to follow instructions and behave like an assistant.
  3. RLHF / RLAIF — humans (or another AI) rate pairs of responses, and the model is updated to prefer the higher-rated style. This is what makes models polite, refusal-aware, and aligned to a usage policy.

When you call Claude or GPT-4, you're talking to the post-RLHF model. The base model still exists under the hood but is rarely exposed.

Why LLMs Hallucinate (And What to Do About It)

A hallucination is a confident statement that's false. It happens because the model is optimizing for plausible next tokens, not for truth. If the training data didn't cover something, the model will still produce something that looks right — it doesn't know what it doesn't know.

Mitigations (in order of impact):

  • RAG — give the model the actual source material so it doesn't have to guess.
  • Citation enforcement — force the model to cite which retrieved chunk supports each claim.
  • Refusal training — instruct the model to say "I don't know" when the context doesn't contain the answer.
  • Lower temperature — for factual tasks, run at temperature 0.
  • Verification step — a second pass that checks the answer against the source.
  • Faithfulness eval — measure hallucination rate continuously in production.

Prompt Engineering — The Useful Bits

  • Be specific. "Summarize" → "Summarize in 3 bullets, max 15 words each, focused on financial risk."
  • Show, don't tell. Few-shot: include 2–5 examples of input → desired output. Often more effective than long instructions.
  • Structure the output. Ask for JSON, a table, or a fixed schema. Easier to parse and validate downstream.
  • Chain-of-thought. "Think step by step before answering" — measurably improves reasoning on hard problems.
  • System prompt = identity + rules. User prompt = the task. Keep them separated.
  • Place key instructions at the end of long prompts (recency bias).
  • Don't argue with the model. If a prompt tweak doesn't fix behavior, change the architecture (add a tool, add retrieval, add validation) — don't keep adding "DO NOT HALLUCINATE" in caps.

Prompt Caching — The Cost Lever Most People Miss

Modern providers (Anthropic, OpenAI) let you cache parts of the prompt that don't change — typically the system prompt and any large reference docs. Cached tokens cost ~10% of normal price and are much faster on subsequent calls.

Anthropic's Contextual Retrieval technique relies on this: you can run an LLM over a 100k-token document chunk-by-chunk cheaply because the document itself is cached. Without prompt caching it'd be ruinously expensive.

Tool Calling (Function Calling) — Giving the LLM Hands

A vanilla LLM can only produce text. Tool calling (also called "function calling") gives it the ability to request that your code execute a function and return the result — turning a text generator into something that can read databases, hit APIs, run calculations, or kick off workflows.

Important: the LLM does NOT execute the tool itself. It outputs a structured request like "call get_balance with account_id='ACC-123'" — your code runs the function and feeds the result back into the next LLM call.

The Tool-Call Loop

1. You define tools (name, description, JSON schema for arguments).
2. You send: user message + tool definitions to the LLM.
3. LLM responds in one of two ways:
     (a) Plain text answer  → done.
     (b) Tool-use request   → "call get_balance({account_id: 'ACC-123'})"
4. Your code executes the requested tool.
5. You append the tool result to the conversation and call the LLM again.
6. LLM either calls another tool or produces the final answer.
7. Repeat until done.

What Makes a Good Tool

  • Clear, specific name. lookup_mortgage_rate beats get_data.
  • Rich description. The LLM picks tools by reading descriptions — vague descriptions = wrong tool calls.
  • Strict argument schema. Use types and descriptions on every parameter. Validate before executing.
  • Atomic. One tool = one logical action. Don't make a "do_everything" tool — the LLM can't reason about it.
  • Idempotent where possible. Tools may be retried; reads are safe to retry, writes need idempotency keys.
  • Honest errors. Return "product code 'XYZ' not found", not a 500. The model uses error text to recover.

Tool Calling — Minimal Example

from anthropic import Anthropic
client = Anthropic()

tools = [{
    "name": "get_account_balance",
    "description": "Return the current balance for a customer account in GBP.",
    "input_schema": {
        "type": "object",
        "properties": {
            "account_id": {"type": "string", "description": "Account ID, e.g. 'ACC-1234'."}
        },
        "required": ["account_id"],
    },
}]

def get_account_balance(account_id: str) -> dict:
    return {"account_id": account_id, "balance": 12450.77, "currency": "GBP"}

messages = [{"role": "user", "content": "What's the balance on account ACC-1234?"}]

while True:
    resp = client.messages.create(
        model="claude-opus-4-7", max_tokens=1024, tools=tools, messages=messages,
    )
    messages.append({"role": "assistant", "content": resp.content})

    if resp.stop_reason != "tool_use":
        break  # final answer

    # Execute every tool the model asked for
    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            result = get_account_balance(**block.input)
            tool_results.append({
                "type": "tool_result", "tool_use_id": block.id,
                "content": str(result),
            })
    messages.append({"role": "user", "content": tool_results})

print(resp.content[0].text)

Patterns & Pitfalls

  • Parallel tool calls. Modern models can request multiple tools in one turn — execute them concurrently for latency wins.
  • Tool injection. Tool results are untrusted input. A malicious doc returned by a search tool could try to rewrite the system prompt. Treat tool output like user input — never execute instructions inside it.
  • Schema drift. If a tool's signature changes, in-flight conversations break. Version your tools.
  • Loops. Without a hop limit, a confused model can call tools forever. Cap iterations (e.g., recursion_limit=12 in LangGraph).
  • Hallucinated tools. Models sometimes invent tools that don't exist. Hard-validate the tool name before executing.
  • Forced tool use. You can require the model to call a specific tool with tool_choice. Useful for structured extraction.

Tool Calling vs Structured Output vs RAG

PatternUse when
Structured outputYou want the LLM to fill a fixed schema from the input it already has. No external action.
Tool callingThe LLM needs fresh data, to do math, or to take an action. Dynamic.
RAGYou always need to ground in a knowledge base; you control retrieval, not the model. Predictable.
Agentic RAGCombine: model decides when and how to retrieve via a search tool. Most flexible, most expensive.

Code Interpreter — Letting the LLM Run Code

A code interpreter (also called "code execution" or "code sandbox") is a special tool that gives the LLM a sandboxed Python environment where it can write code, execute it, see the output, and iterate. It's a single tool with a huge effect on capability.

Why it matters: LLMs are bad at doing math but good at writing the math. A vanilla LLM might fluff "23,481 × 1.0875 across 12 monthly payments" — but if it can write Python and see the result, it nails it every time. Same for sorting, filtering, regex, statistics, parsing JSON/CSV, generating charts.

What It's Good For

  • Numerical work — interest calculations, amortization schedules, fee breakdowns, statistics over a dataset.
  • Data wrangling — clean a CSV, pivot a table, filter rows, join two files.
  • Chart generation — produce matplotlib/seaborn charts and return as images.
  • Code transformation — refactor a snippet, run it to verify, return both code and output.
  • File processing — extract from PDFs, parse Excel, regex over logs.
  • Verification — write a quick test to check whether the model's own reasoning was correct.

How It Works (Provider Implementations)

  • OpenAI — built-in code_interpreter tool in the Assistants/Responses API. Sandboxed Python, ephemeral file workspace, automatic.
  • Anthropic — exposes a code execution tool you can enable; Claude writes code, the platform runs it, results return as a tool result.
  • Roll-your-own — define a run_python tool that executes inside a hardened sandbox (Docker container, gVisor, Firecracker, or services like E2B / Modal). You own the security boundary.

Security — The Non-Negotiables

  • Sandbox isolation — never run model-written code on a host with real credentials, network, or filesystem.
  • No outbound network by default; allowlist if needed.
  • Resource caps — CPU, memory, wall-clock time, output size.
  • Ephemeral filesystem — destroyed at the end of the session.
  • Input scrubbing — model-written code can be a prompt-injection vector if a malicious doc says "...and now write code that exfiltrates X." Treat it as untrusted by default.

Banking Examples

  • Quote builder — RM asks "what would the monthly repayment be on £350k over 25 years at our current 5y fixed rate?" The model fetches the rate via a tool, then runs the math via code interpreter, returning a breakdown.
  • Portfolio analyser — uploaded CSV of holdings → code interpreter computes exposures, top concentrations, and a chart.
  • Reconciliation helper — two extracts, find mismatches, output the diff with a sample.
Interview point: "I add code interpreter when the workflow needs precision arithmetic, data manipulation, or visualization. It pairs well with RAG — RAG fetches the facts, code interpreter does the computation. But the sandbox security model has to be airtight before MRM signs off."

Skills — Packaging Instructions and Capabilities

A Skill is a reusable bundle of instructions, examples, and optional resources (scripts, templates, references) that an agent loads only when relevant to the current task. Think of it as a folder of "how to do X" that the model pulls into context on demand instead of stuffing every possible instruction into the system prompt.

Skills are core to Claude Code, the Claude Agent SDK, and Anthropic's Skills product — but the pattern applies to any agent framework. The idea: keep the system prompt small and stable; let the agent discover the right skill for the task.

Anatomy of a Skill

skills/
└── refund-policy/
    ├── SKILL.md           ← name, when-to-use trigger, instructions
    ├── examples/
    │   ├── partial.md
    │   └── full-refund.md
    ├── scripts/
    │   └── calculate_refund.py
    └── references/
        └── policy-2026.pdf

SKILL.md always starts with frontmatter:

---
name: refund-policy
description: Use when a customer requests a refund or chargeback.
  Covers eligibility, timelines, and approval thresholds.
---

# Refund Policy Skill

## When to use
- Customer mentions refund, chargeback, money back, ...
- Claim disputes against a completed transaction.

## Steps
1. Verify the transaction status with `lookup_transaction`.
2. Check eligibility against rules in references/policy-2026.pdf.
3. If amount < £500, draft refund confirmation. If ≥ £500, escalate to manager.

## Examples
See examples/partial.md and examples/full-refund.md.

How a Skill Gets Used

  1. At session start, the agent harness loads only the metadata (name + description) of every available skill — cheap.
  2. When the user asks something, the agent reads the descriptions and decides "this looks like a refund question."
  3. The agent invokes the skill, which loads the full SKILL.md into context — instructions, examples, and pointers to scripts/references.
  4. The agent follows the skill, calling tools or reading references as the skill directs.

Skills vs Tools vs RAG vs System Prompt

MechanismWhat it carriesLifetime
System promptIdentity + always-applicable rulesEvery turn — keep it small
SkillTask-specific instructions, examples, references — plus when to use themLoaded on demand for the relevant turn
ToolAn executable function (code that runs)Available every turn; called when needed
RAG contextSnippets from a knowledge baseInjected per query based on similarity

Mental model: Tools are verbs (actions). Skills are playbooks (how-tos). RAG is reference material (facts). A mature agent uses all three: it follows a skill, which tells it to call certain tools, which retrieve from a RAG store.

Why Skills Beat a Mega System Prompt

  • Context efficiency. Skills only load when relevant — no paying for refund instructions on a mortgage question.
  • Modularity. Each skill owns its domain. SMEs can update one skill without touching the rest.
  • Versioning. Skills live in git; rollouts and rollbacks are diffs.
  • Discoverability. Adding capability = adding a skill folder. No prompt-engineering tax.
  • Composability. Skills can reference other skills ([[other-skill]]). Capability grows like a library, not a monolith.

Banking Example — Skill Library for the Client Pathways Assistant

skills/
├── kyc-verification/        # AML/KYC checks during onboarding
├── product-recommendation/  # match products to client segment
├── complaint-handling/      # FCA-compliant complaint flow
├── rate-quotation/          # produce a rate quote with disclaimers
├── escalation-rules/        # when and to whom to escalate
└── disclosure-templates/    # regulatory disclosures by product/region

A relationship-manager query about a complaint loads complaint-handling; a rate question loads rate-quotation. The system prompt stays a few hundred tokens and never has to know about every domain up front.

Interview soundbite: "Skills are how I keep agent system prompts from rotting into 5,000-line monsters. The agent discovers the right playbook for the task at hand — and SMEs maintain those playbooks like documentation, in version control."

If you remember nothing else: LLMs are next-token predictors with a context window and no memory. They have no truth check. Everything we build around them — RAG, agents, evals, guardrails — exists to make a probabilistic next-token machine behave reliably enough for production.
