L3 The Model Landscape: Providers, Modalities & How to Choose

"Which LLM should we use?" is the most common architecture question you'll get asked. This section gives you a working map of the major providers, what multimodal means, and how to make the choice defensibly.

What is "Multimodal"?

A modality is a kind of input or output — text, image, audio, video. A multimodal model can natively handle more than one. The key distinction:

  • Multimodal input — model can see images or hear audio along with text. Lets you ask "what's on this scanned mortgage form?" or "transcribe and summarize this client call."
  • Multimodal output — model can produce non-text, e.g., generate an image, speak a response, draw a chart.
  • Native multimodal — single model trained jointly on all modalities (GPT-4o, Gemini). Reasons across modalities (e.g., understands a chart and the surrounding text together).
  • Pipeline multimodal — separate models stitched together (Whisper → text-LLM → TTS). More flexible, often cheaper, less coherent across modalities.

Banking applications: reading scanned ID documents (KYC), analyzing trade confirmation PDFs with tables and stamps, transcribing and summarizing advisor calls, extracting data from cheque images, OCR-free invoice processing.

The Major Providers — Detailed Comparison

Anthropic (Claude)

Models: Claude Opus (top capability), Claude Sonnet (workhorse), Claude Haiku (fast/cheap). Current family is 4.x.

  • Strengths: Best-in-class for long-context comprehension (200k tokens), nuanced writing, code generation, tool use accuracy. Strong safety alignment. Constitutional AI training. Excellent at following complex instructions. Prompt caching at ~10% of input cost is a big lever.
  • Weaknesses: No native image generation. Slightly behind on raw multimodal vs Gemini. Higher per-token cost than open-weight options.
  • Best for: Enterprise assistants, code agents (Claude Code), regulated workflows where reliability and instruction-following matter, long-document analysis, RAG over large knowledge bases.
  • Banking fit: Excellent. Strong on reasoning over policy docs, refusal behavior on out-of-scope queries, citation enforcement.

OpenAI (GPT, o-series)

Models: GPT-4.1 / GPT-4o (general), o3 / o4-mini (reasoning), GPT-4o-mini (cheap), Whisper (ASR), DALL·E 3 (image gen), TTS.

  • Strengths: Broadest ecosystem and tooling, best image generation in a major API, native voice, strong reasoning models (o-series). Largest developer community → most examples and integrations. Full multimodal stack from one vendor.
  • Weaknesses: Pricing on top models can be high. Periodic API instability. More restrictive content moderation that can block legitimate use cases. Reasoning models hide their chain-of-thought (less debuggable).
  • Best for: Consumer-facing chatbots, voice agents, image-heavy applications, workflows that benefit from o-series reasoning (math, code, planning).
  • Banking fit: Good, but Microsoft Azure OpenAI is the typical procurement path for enterprise (data residency, BAA-style contracts).

Google (Gemini)

Models: Gemini 2.0 Flash / Pro / Ultra, Gemini Thinking, Imagen, Veo (video).

  • Strengths: Massive context window (1M+ tokens, experimental 2M). Native multimodal across text, image, audio, video — strongest of any provider on video understanding. Tight integration with Google Cloud and Workspace data. Often cheapest at the top tier.
  • Weaknesses: Quality has historically been less consistent than Claude/GPT on hard tasks. SDK and tooling less mature. More frequent API and product reshuffles.
  • Best for: Massive-context use cases (entire codebase or document set in one prompt), video analysis, GCP-native shops, multimodal agents.
  • Banking fit: Strong if you're on GCP. Vertex AI gives enterprise controls. Useful for analyzing very large regulatory document corpora in single passes.

Meta (Llama)

Models: Llama 3.x (8B, 70B, 405B), Llama 3.2 Vision, Llama Guard.

  • Strengths: Open-weight — you can run it on your own infrastructure. Best open-source model family. Free to use commercially (with license terms). Strong fine-tuning ecosystem. Good multilingual coverage.
  • Weaknesses: 405B model needs serious GPU infrastructure to serve. Smaller variants are noticeably below frontier. Tool-use accuracy lags top closed models. You own all the ops burden.
  • Best for: On-prem deployments where data can't leave (regulatory or sovereignty), customized fine-tunes, cost optimization at very high volumes, privacy-sensitive workloads.
  • Banking fit: Common for use cases where data residency or model risk management forbids third-party APIs. Often paired with a closed model for tasks where Llama isn't strong enough.

Mistral AI

Models: Mistral Large, Mistral Small, Codestral (code), Mixtral (MoE), open-weight variants.

  • Strengths: European provider — important for EU data sovereignty. Open and closed offerings. Strong efficiency (Mixture-of-Experts). Solid multilingual European language coverage.
  • Weaknesses: Generally one tier below Claude/GPT/Gemini at the top end. Smaller ecosystem.
  • Best for: EU-regulated workloads (GDPR, EU AI Act), French/German/Italian/Spanish use cases, organizations wanting a non-US vendor.
  • Banking fit: Strong for European banks; often the political/regulatory choice over US providers.

Cohere

Models: Command R / R+ (chat), Embed v3, Rerank.

  • Strengths: Best-in-class embedding and reranking models — many production RAG stacks use Cohere here even with a different chat model. Enterprise focus, strong RAG-tuned models with citations baked in. Available on AWS / Azure / OCI / on-prem.
  • Weaknesses: Chat models are good but not frontier. Smaller mindshare than the big three.
  • Best for: Enterprise RAG (especially Embed + Rerank), workflows needing built-in citation, customers wanting deployment flexibility.
  • Banking fit: Excellent — built specifically for enterprise; supports private deployments and is commonly used in regulated industries.

DeepSeek

Models: DeepSeek-V3, DeepSeek-R1 (reasoning), DeepSeek-Coder.

  • Strengths: Strong reasoning at extremely low cost. Open-weight. R1 was the first open-weight model to approach OpenAI o1 on benchmarks.
  • Weaknesses: Chinese provider — many Western enterprises won't use the hosted API on data-residency or geopolitical grounds. Hosted version logs queries.
  • Best for: Cost-sensitive R&D, self-hosted reasoning workloads, regions where it's an acceptable provider.
  • Banking fit: Limited in Western banks via the hosted API. Some interest in self-hosted variants.

xAI (Grok)

Models: Grok 2, Grok Vision.

  • Strengths: Real-time access to X data; less restrictive content policies; competitive on benchmarks.
  • Weaknesses: Smaller ecosystem; less proven enterprise track record; brand baggage.
  • Best for: Use cases needing X / social-media data; lightly-regulated experimentation.
  • Banking fit: Rare. Most banks won't approve it for regulated workflows.

Open-Source Stack (HuggingFace ecosystem)

Models: Qwen, Phi, Gemma, BGE, Nomic, thousands more.

  • Strengths: Total control, no per-token cost, full transparency, fine-tune freely, run anywhere including air-gapped environments.
  • Weaknesses: You operate it. GPU costs, scaling, monitoring, security patching all on you. Smaller models lag frontier.
  • Best for: Embedding/reranking (very competitive open-source), specialized fine-tunes, on-prem requirements, high-volume cheap inference.
  • Banking fit: Common for embedding models (BGE-m3, e5-large) and small specialized tasks; less common as the primary chat model.

Image Generation Models — Text-to-Image Deep-Dive

Image generation models turn a text prompt into a pixel image. They are not LLMs — they're diffusion models (or, for some newer ones, autoregressive transformers). Worth understanding because they appear in many enterprise GenAI initiatives (marketing, KYC simulation, design assist) and have very different operational characteristics from text LLMs.

How Diffusion Works (Plain English)

  1. During training, the model is shown images that have been progressively corrupted with random noise.
  2. It learns to reverse the noise — given a noisy image, predict what noise to remove.
  3. At generation time, you start with pure random noise and the model "de-noises" step by step (typically 20–50 steps), guided by your text prompt via a text encoder (CLIP/T5).
  4. After all steps, the noise has been shaped into a coherent image matching the prompt.

The Major Players

  • DALL·E 3 (OpenAI) — strong on prompt adherence and text rendering inside images. API access, content moderation built in.
  • Imagen 3 (Google) — high photorealism, native via Vertex AI. Strong on faces and detail.
  • Stable Diffusion (Stability AI) — open-weight. Run anywhere. Massive ecosystem of fine-tunes, LoRAs, and ControlNets. The default for self-hosted.
  • Flux (Black Forest Labs) — recent open-weight family, currently best-in-class for open generation. Pro variant via API.
  • Midjourney — best aesthetic quality "out of the box," strong artistic style. Discord-only historically; API increasingly available.
  • Ideogram — specializes in correct text rendering inside images.
  • Adobe Firefly — trained on licensed data — important for enterprise IP/legal compliance.

Important Capabilities Beyond "Text → Image"

  • Inpainting — edit a specific region of an image while keeping the rest.
  • Outpainting — extend an image beyond its original borders.
  • Image-to-image — transform an existing image guided by a prompt.
  • ControlNet — condition generation on a structural input (pose skeleton, edges, depth) to control composition precisely.
  • LoRAs / fine-tunes — small weight overlays to teach a model a specific style, face, or product.
  • Reference images — newer models accept a reference image to maintain identity or style across generations.

Enterprise Concerns (Especially in Banking)

  • Copyright & training data. Some providers (Adobe Firefly, Getty) use only licensed data — the right answer for regulated marketing. Open Stable Diffusion has murkier training-data lineage.
  • Brand safety. Output filtering for nudity, violence, public figures, logos. All major providers offer this; the failure modes differ.
  • Deepfake risk. Generating realistic faces of real people is a huge reputational risk. Most providers block public figures by name; you should add identity-detection on top.
  • C2PA / provenance. Industry-standard metadata for "this image was AI-generated" — increasingly required for marketing, regulated comms.
  • Cost. ~$0.04–$0.08 per image at API pricing. Self-hosted Stable Diffusion is cheaper at volume but you eat GPU costs.

Banking / Finance Applications

  • Marketing creative — campaign visuals, ad variants, A/B tests.
  • Document templates — generate illustrative diagrams for client comms.
  • UI mockups for digital products without a designer in the loop.
  • Synthetic ID document samples for KYC model training (carefully, with provenance).
  • Not for: anything client-facing without legal review of the training-data provenance.
Quick mental model: LLMs ≈ probabilistic text predictors. Diffusion models ≈ probabilistic noise removers. Different math, different failure modes, different ops.

VLMs (Vision-Language Models) — Multimodal RAG

A VLM is a model that natively understands both text and images. Same transformer architecture as an LLM, but with a vision encoder bolted in front so image patches and text tokens flow through the model together. Examples: GPT-4o, Claude (vision), Gemini, Llama 3.2 Vision, Qwen2-VL.

Why this matters for enterprise RAG: most enterprise documents are not pure text. Mortgage forms have stamps and signatures. Trade confirmations have tables. Pitchbooks have charts. KYC docs are scans. Traditional text-only RAG throws all that away at parse time — VLMs let you keep it.

Text-Only RAG vs Multimodal (Vision) RAG

AspectText-only RAGVision RAG (with VLM)
IngestOCR + layout parsing → textRender each page as an image; optional text extract
Tables & chartsOften broken or lostPreserved — model sees them visually
IndexText embeddings of chunksImage embeddings of pages (e.g., ColPali) + optional text
RetrievalMatch query embedding to text chunksMatch query embedding to page images (or both)
GenerationLLM reads retrieved textVLM reads retrieved page images directly
CostLower — text is cheapHigher — images use more tokens

Two Architectural Patterns

  • (1) Pre-extract pattern — at ingest, run a VLM to convert images/tables/charts to rich text descriptions, then index that text. Cheaper at query time; lossy because you fix the description up front.
  • (2) Native vision pattern — index the page images themselves (e.g., with ColPali — a vision retriever that produces multi-vector page embeddings). At query time, retrieve pages and pass the actual page images to a VLM. Most accurate, more expensive at query time.

Key Tools & Models for Multimodal RAG

  • ColPali / ColQwen — vision retrievers that beat text pipelines on document benchmarks (DocVQA-style). They produce a multi-vector representation per page, with late interaction at retrieval.
  • VLMs for generation — Claude, GPT-4o, Gemini, Qwen2-VL are all strong. Pick by capability + cost as before.
  • Document parsers — Unstructured, LlamaParse, AWS Textract, Azure Document Intelligence, Docling — for the pre-extract pattern.
  • Open VLMs — Llama 3.2 Vision, Qwen2-VL, InternVL — for self-hosted multimodal RAG.

When to Use Multimodal RAG (Banking)

  • Trade and confirmations — tables and stamps that text extraction mangles.
  • Mortgage and loan files — multi-page scans with handwriting, signatures, pre-printed forms.
  • Regulatory filings — dense tables, footnotes, complex layouts (10-Ks, prospectuses).
  • Pitchbooks & research — charts and diagrams contain the actual insight.
  • KYC documents — IDs, utility bills, photos.

Caveats

  • Cost. A single page image can be 1k–2k tokens. Retrieving 5 pages × 1.5k tokens is 7.5k input tokens per query — meaningful at scale.
  • Latency. Vision tokens are slower to process. Plan for higher p95.
  • Eval is harder. Faithfulness on a chart is ambiguous; build domain-specific deterministic checks (e.g., "does the cited number match the chart's value?").
  • Hybrid usually wins. Index both page images and extracted text; route at query time.
Interview point: "If the corpus is mostly clean text, stick with classic RAG. If it's PDFs with tables, charts, scans, or stamps, native vision RAG (ColPali + a VLM) is now the right default — it routinely beats heroic OCR pipelines on document benchmarks. Hybrid retrieval over both representations is the production pattern."

Cloud-Hosted Variants & Enterprise Procurement

In regulated industries, you rarely buy the model directly from the provider — you go through a hyperscaler that wraps it with enterprise controls (private networking, customer-managed keys, data residency, BAA / DPA contracts, no training-on-your-data guarantees).

  • AWS Bedrock — Claude, Llama, Mistral, Cohere, Titan, Stable Diffusion in one API. The default in AWS shops.
  • Azure OpenAI Service — GPT family with Microsoft's enterprise wrapping. Default in Microsoft shops.
  • Google Vertex AI — Gemini, plus Anthropic and open models. Default in GCP shops.
  • Anthropic on AWS / GCP — Claude is available natively on both, often the simplest enterprise procurement path.
  • Databricks Mosaic AI / Snowflake Cortex — host models next to your data. Increasingly popular for regulated analytics.

Why this matters: a model risk management (MRM) team will not sign off on calling api.openai.com directly from a banking workload. Going through Bedrock / Azure / Vertex with private networking and contractual non-training is usually mandatory.

How to Choose a Model — A Decision Framework

Apply these filters in order. Stop at the first one that constrains your choice meaningfully.

  1. Compliance & data residency. Where can the data go? On-prem only? EU only? This often eliminates 80% of options up front.
  2. Capability needed. Is this a frontier-reasoning task or a simple classification? Don't pay Opus prices for a job Haiku can do.
  3. Modalities. Need vision? Audio? Video? Image generation?
  4. Context window. How much do you actually need to fit? Most use cases are fine on 32k–128k.
  5. Latency budget. Real-time chat (<1s first token) vs batch (<30s)? Smaller models or self-hosted may be required.
  6. Cost at expected volume. Multiply tokens × $/token × QPS × seconds in a month. Numbers get scary fast — model-route cheap-first.
  7. Vendor risk. Single vendor = lock-in. Abstract behind your own interface; keep an exit plan.
  8. Evals on your data. Last and most important: run your own gold set. Benchmarks lie; your eval doesn't.

Model Routing — The Pattern That Saves Money

In production, you rarely use one model for everything. Model routing sends easy requests to a small/cheap model and escalates only the hard ones to a frontier model.

  • Tier-by-task — classification → small model; final answer → frontier model.
  • Confidence-based escalation — small model answers; if confidence (or a checker) is low, escalate.
  • Fallback chain — try fastest first, fall back to slower on failure or low quality.
  • Specialty routing — code questions → Claude/Codestral; image questions → multimodal; reasoning → o-series.

Real-world impact: a well-tuned router can cut LLM spend by 60–80% with no measurable quality loss. This alone often pays for the engineering effort to build the abstraction.

Encoder & Retrieval Models — BERT Family & Beyond

RAG quality is often decided by the encoder stack, not the chat LLM. These are the models behind embeddings, reranking, and hybrid search — know how they relate.

The BERT Family — Where It All Started

BERT (Bidirectional Encoder Representations from Transformers) reads text in both directions and outputs contextual token embeddings. It is an encoder-only model — great for understanding, not for generating text. Most modern embedding and reranking models are BERT descendants.

Decoder LLMs vs Encoder Models — Don't Confuse Them

  • Encoder-only (BERT, embedders, rerankers) — read input, output vectors or scores. Used for search, classification, NER. Cannot generate text.
  • Decoder-only (GPT, Claude, Llama) — predict next token. Used for chat, agents, generation. Embeddings from these are possible but not their primary design.
  • Encoder-decoder (T5, BART) — seq2seq; summarization, translation. Less common in modern RAG stacks.
  • Interview line: "My chat model is Claude; my retrieval stack is bge-m3 + Cohere Rerank — different models for different jobs."

Small Models — When & Which Ones

Small models (<3B params for LLMs, <500M for encoders) are not "worse LLMs" — they're the right tool for high-volume, low-latency, or on-prem tasks where frontier capability isn't needed.

Small LLMs — Routing, Rewrite, Classification

ModelParamsBest for
Claude Haiku / GPT-4o-mini—API-tier fast models: query rewrite, intent classification, simple extraction
Llama 3.2 1B / 3B1–3BOn-device or VPC chat for routing; not for regulated final answers alone
Phi-3-mini / Phi-3.5-mini3.8BStrong reasoning for size; good for structured extraction on CPU/GPU
Gemma 2 2B2BGoogle open small model; efficient on single GPU
Mistral 7B / Mixtral 8×7B7–47B"Small" for self-hosted chat; MoE variant is fast at inference

Small Encoders — Embedding & Reranking

  • all-MiniLM-L6-v2 (22M) — 384-d; baseline for prototyping; ~50k sentences/sec on GPU.
  • bge-small-en-v1.5 (33M) — best small English embedder on MTEB; self-host for zero API cost.
  • e5-small-v2 (33M) — Microsoft; requires query: / passage: prefixes.
  • ms-marco-MiniLM-L-6-v2 (22M) — cross-encoder reranker; rerank 50 pairs in <100ms on CPU.
  • bge-reranker-base (278M) — step up when MiniLM reranker quality isn't enough.

Decision Guide

  1. Task is binary classification / routing? → Small LLM or even a fine-tuned DistilBERT.
  2. Task is semantic search at >100 QPS? → Small bi-encoder self-hosted; validate recall on gold set.
  3. Task is reranking top-50? → ms-marco-MiniLM on CPU; upgrade to bge-reranker if metrics demand.
  4. Task is customer-facing regulated answer? → Frontier LLM (Claude Sonnet+), not a 1B model.
  5. Always: run your eval harness — a 33M embedder often beats a 7B LLM at retrieval because retrieval is an encoder job, not a generation job.
